# ADR-0043: 基本的なプロシージャルテクスチャ(checker/voronoi/brick)をネイティブ追加する

- Status: accepted
- Date: 2026-07-07

## Context

「サーフェイスにテクスチャを追加したい。基本的なプロシージャルテクスチャは
用意しておきたいが、これもFFIでシェーダーを書くべきか」という相談があった。

この言語の核心設計(`Shape = {dist, colour}`、`colour` は
`(座標, 時間) → 色` の純粋関数)からすると、テクスチャは特別な仕組みでは
なく「`fill` に座標依存の色(Field)を渡す」だけで実現できる。実際
`noise`/`fbm`/`stripes` が既にこの形でネイティブ提供されている。FFI
(`wgsl`/`glsl` ブロック、ADR-0011)は「一回限りの特殊なシェーダーを
書きたい」ためのエスケープハッチとして既にあり、汎用的に何度も使う
プリミティブをそこに押し込めると、次元多相やリテラルのuniform昇格
(ADR-0008)の恩恵を受けられず、使うたびに書き直しが必要になる。

## Decision

`stripes`/`noise`/`fbm` と同じ流儀(`stdlib/noise.ts` にネイティブの
`Field` ビルトインとして追加)で、次の3つを追加した:

- **`checker s`**(`Field d Float`、2D/3D自動判別): セルサイズ `s` の
  チェッカーボード。`floor(p/s)` の各成分の和を2で割った余り。既存の
  `fmod` ライブラリ関数だけで組めるので、新規WGSL関数は不要
- **`voronoi s`**(`Field d Float`、2D/3D自動判別): セルサイズ `s` の
  ボロノイ/セルラーノイズ(F1、最近傍のジッタ格子点までの距離、
  0..1にクランプ)。3x3(2D)/3x3x3(3D)の近傍総当たりが要るため、
  `wgsl.ts` に `voronoi2`/`voronoi3` を新設。ジッタ用に `hash33`
  (vec3→vec3)も新設し、ADR-0039のbitcast+MurmurHash3(fmix32)方式を
  そのまま踏襲した(`hash22`/`hash13`と同じ形で座標成分ごとに
  `hashMix`してXOR合成)
- **`brick s`**(`Field2 Float`、`stripes`と同じく2D専用): セル幅 `s`、
  2:1のレンガ、1段おきに半幅ずらす定番パターン。IR合成のみで実装

`noise` は既存の `noise`/`fbm` がそのまま使えるため新規追加なし。

いずれも `fill` に直接渡すだけで使える(`fill` は Field を受け取ると
`asColor` でグレースケール昇格するので、`fill (checker 0.2)` のように
書ける)。3D図形にも使え、`checker`/`voronoi` は本物の3D距離場テクスチャ
(世界座標をそのままサンプルする)になる一方、`brick` は `stripes` と
同じくxy平面への投影になる(次元の整合性は`fill`側でチェックしていない
——これは`stripes`の既存の挙動と同じで、新しい制限ではない)。

## Consequences

- ✅ `fill (checker 0.2)`/`fill (voronoi 0.2)`/`fill (brick 0.25)` と
  書くだけで使える。実機で確認済み: 2D(円に直接)・3D(球面に投影、
  voronoiは真の3D距離場としてシームレスに)の両方で正しく描画される
- ✅ 次元多相(checker/voronoi)・リテラルのuniform昇格(sの値をスクラブで
  即座に変えられる)など、ネイティブField共通の恩恵をそのまま受ける
- ✅ 突飛な柄(汎用UVマッピング・法線マップ的な効果・特殊な合成)は
  引き続きFFI(`wgsl`ブロック)の領分として残る。今回追加した3つは
  「よく使う基本形」に限定し、無限に広がりうる柄のカタログを
  ネイティブ側で追いかけようとはしない(ADR-0013の「組合せ展開を
  避ける」という方針と整合)
- ⚠️ `brick` を3D図形に使うと(`stripes`と同じく)xy平面への投影になる。
  意図しない見た目になりうるが、コンパイルエラーにはしていない
  (既存の`stripes`の挙動を変えない選択)
