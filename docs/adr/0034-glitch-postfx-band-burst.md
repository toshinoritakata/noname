# ADR-0034: `glitch` は走査線帯 × 離散時間ステップのバースト発火として実装する

- Status: accepted
- Date: 2026-07-07

## Context

既存の postfx(`fade`/`zoom`/`bloom`/`chromatic`/`grain`/`vignette`)に、
デジタルグリッチ/データモッシュ系の表現が無かった。ライブビジュアルの
定番エフェクトの一つ。

単純に「毎フレーム・全画面にランダムなノイズ/ずれを乗せる」実装は、
見た目が単なる高周波ノイズになり glitch らしいバースト感が出ない
(`grain` と代わり映えしない)。実際のグリッチ映像(データ破損・同期外れ)は
**特定の走査線帯が、特定の瞬間だけ**ずれる、という「疎で離散的な発火」が
特徴。

## Decision

`glitch k x`(`src/compiler/stdlib/postfx.ts`)を次の式で実装する:

```
band = floor(p.y * 60)          -- 画面を60本の走査線帯に分割
tick = floor(time * 10)         -- 時間を1/10秒刻みの離散ステップに分割
seedA = band*13.37 + tick*7.91
seedB = band*3.1  + tick*2.7
amt   = hash11(seedA)           -- 帯×ステップごとに決まる乱数(2つ)
dir   = hash11(seedB)
active   = step(1 - k*1.2, amt) -- k が大きいほど発火する帯が増える(0か1)
shiftX   = (dir*2-1) * k * 0.25 * active  -- 発火中の帯だけ水平にずらす
chromaOff = k * 0.015 * active            -- 発火中はさらにRGBを少しだけ割る
```

`hash11`(既存のWGSLライブラリ関数、`hash`/`noise` 系が既に使っている)を
`band`/`tick` から決まるシードで呼ぶことで、**同じ帯は同じステップの間
ずっと同じずれ方をする**(毎フレーム暴れない)。ステップが変わった瞬間
だけ別の帯が発火し、ずれ方も変わる ―― これが「バースト」感を生む。

`chromatic`(既存)と同じく、ずれた位置を中心に R/G/B を少しだけ違う
x座標でサンプルすることで、発火中の帯だけ色収差も乗る(発火していない
帯は `chromaOff=0` になるので通常の `chromatic` 効果は乗らない)。

## Consequences

- ✅ `k` 1つで「発火頻度・強さ」の両方が連動して増える、扱いやすいパラメータ
- ✅ 既存の `img.fn` を3回サンプルする実装(`chromatic` と同じパターン)を
  そのまま流用でき、新しいIRノード種別もテクスチャも増えていない
- ✅ 実機で確認: 8×8グリッドに掛けると、時間経過で異なる帯が入れ替わり
  立ち替わり発火し、単なるノイズではなく明確な「グリッチ」に見える
- ⚠️ 帯の本数(60)とステップ頻度(10Hz)は固定値。将来的にパラメータ化
  (`glitch {bands, rate, k}` 等)する余地はあるが、まずは単一引数の
  シンプルな形で様子を見る
- ⚠️ 3回のテクスチャサンプル(`chromatic` と同様)を伴うため、既に重い
  シーンに追加するとコストが乗る
