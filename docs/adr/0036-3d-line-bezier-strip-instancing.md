# ADR-0036: 3D の line/bezier を sprite と同じ深度テストなしビルボードで instanced 描画する

- Status: accepted
- Date: 2026-07-07

## Context

[[ADR-0016]] で2Dの `line`/`bezier` は三角形ストリップの instanced 描画に
対応したが、3Dは通常のSDFレイマーチのみで、`scatter` した3D線分は
[[ADR-0035]] の警告診断が実際に検出する「見えない性能崖」の対象だった
(O(n) の SDF ループにフォールバックし、n が大きいと重くなる)。

3Dでの instanced 描画には、2Dストリップにはなかった課題が2つある:

1. **幅方向をどう決めるか**: 2Dは進行方向に垂直な1方向(法線)しかないが、
   3Dでは進行方向に垂直な方向が無限にある。カメラに正対する「リボン」に
   するには、視線方向も使って幅方向を一意に決める必要がある
2. **レイマーチ結果との合成**: 3Dの line/bezier は他のSDF図形と同じ
   シーンに属しうる(例: 3Dオブジェクトの周りにワイヤーフレームを配置)。
   正しい遮蔽(手前の不透明物体の後ろに隠れる)には深度バッファが要る

(2)については、[[ADR-0014]] の sprite(点状パーティクルのinstanced描画)が
既に同じ問題に直面し、「深度テストなし、加算ブレンドのみ、カメラ背後は
クリップ座標を画角外に押し出して簡易カリング」という割り切りを採用して
いた。3Dストリップも同じ割り切りを踏襲する — 一貫性があり、実装量も
最小で済む。

## Decision

`VShape` に `strip3D`(単項マーカー)/`strip3Batches`(集約後バッチ)を
追加し、`strip2D`/`stripBatches` と全く同じ規約で扱う: `line`/`bezier`
が3D版で起点を置き、`outline` が幅、`fill` が色を積む。`move` は(2Dの
strip2Dと同じく)対象外(安全にSDFへフォールバック)。`glow` は sprite と
同じ理由で対応する(明るさ変換は座標に依存しないので、`strip3D.colour`
にも同じ boost 式を適用できる — 2Dの strip2D では未対応のままなのと対照的)。

**幅方向の決定**(課題1): 頂点シェーダで `right = normalize(cross(tangent, viewDir))`
とする。`tangent` はベジエの接線(2Dストリップと同じ2次ベジエの微分)、
`viewDir` はカメラ位置から現在点への視線ベクトル。接線と視線がほぼ平行な
縮退ケースはカメラの `right` ベクトルにフォールバックする。これにより
リボンは常にカメラの方を向く。

**レイマーチとの合成**(課題2): sprite と同じく深度テストを行わない。
`loopShape` は dist を定数 `+∞` にすり替え、実体は新設の `strip3d` パス
(sprite と同じ「フレームに1回のデータテクスチャ+instanced描画」構成、
ブレンドも sprite と同じ加算rgb+maxアルファ)がレイマーチの出力テクスチャに
直接重ね描きする。

`liftDist`(ADR-0027)の必須キーに `strip3D` を追加し、`outline`/`distort`/
`warpValue`/`timeWarp` の呼び出し側に明示させた。`shapeUnion`/`cut`/`inter`/
`selectValue`/`mixValue` の `spriteBatches`/`stripBatches` マージにも
`strip3Batches` を同様に加えた(ADR-0027の規律をそのまま延長)。

## Consequences

- ✅ `scatter N (\i -> line/bezier |> outline w |> fill c)` を3Dシーンで
  使っても、ADR-0014の粒子と同じ O(1) の instanced 描画になる。実機で
  確認済み(200本のベジエ曲線が正しくカメラ向きのリボンとして描画され、
  時間とカメラの orbit 回転で正しくアニメーションすることをスクリーン
  ショットで確認)
- ✅ ADR-0035 の警告診断が正しく連動する: 3D line/bezier を `scatter` して
  対象のチェーン(`outline |> fill`)から外れる書き方(`rot` を挟む等)を
  すると警告が出ることをテストで確認。対象のチェーンでは警告が出ない
- ✅ sprite と同じ実装パターン(データパス+instanced描画)を再利用したので、
  新しい仕組みは「幅方向の計算」と「4テクスチャへの制御点パッキング」だけ
- ⚠️ sprite と同じ割り切り: 深度テストをしないため、不透明な物体の手前に
  あるはずのストリップが背後に描かれる(あるいはその逆)場合がある。
  「壊れても絵になる」方向の判断(ADR-0014と同じ)
- ⚠️ `move` は対象チェーンに含まれない(2Dのstrip2Dと同じ既存の制限を
  そのまま踏襲。将来必要になれば `move` の3次元ケースだけ制御点を
  追従させる特別扱いを追加できる)
