# ADR-0016: 2D の line/bezier は SDF ではなく三角形ストリップでラスタライズする

- Status: accepted
- Date: 2026-07-05

## Context

ADR-0015 で line/bezier を「距離ゼロの曲線+outline」として SDF 図形に追加したが、
ユーザーからの指摘で「SDF をメインに置くのはパフォーマンス上問題がある」という
再考を受けた。ADR-0014(パーティクルの CSG ループ問題)と全く同じ根本原因が
line/bezier にも当てはまる: 細い形状を per-pixel の距離場評価+march で扱うのは、
形状の細さに見合わない計算コスト(march 段数 × 評価コスト)を強いる。

線・曲線は元来ラスタライズ(パスを幅方向に押し出した帯=strip polygon)が
自然な表現であり、march が原理的に不要になる。3D での他 SDF 形状との
オクルージョン(深度テスト)は別途 深度バッファの追加が要るため、まず
**2D 限定**でこの最適化を実装する(3D は need に応じて再訪)。

## Decision

`line`/`bezier` が 2D(vec2 引数)のときに限り、ADR-0014 と同型の**マーカー伝播+
専用パス**方式で instanced strip 描画に切り替える。

- `VShape.strip2D`: `line`/`bezier` が起点を置く軽量マーカー
  (`{p0, p1, p2, width, colour}`)。`line` は「制御点 p1 = 中点」の**退化ベジエ**
  として扱い、頂点シェーダのコード経路をベジエと完全に共通化する
- `outline w` が `width` を、`fill c`(場でない色のときのみ)が `colour` を
  設定して引き継ぐ。それ以外の合成子(`move`/`glow`/`distort` など)は
  引き継がない(= 安全に SDF フォールバック、ADR-0014 と同じ設計)
- `scatter` の `loopShape` が `strip2D` を検出したら CSG ループを作らず、
  `StripBatchSpec`(位置3点+幅+色、loopi にのみ依存)を返す。`dist` は
  定数 `+∞` にすり替える(既存の `<over>`/`toImage` のフラット化ロジックが
  そのまま「見えない」を導出する。sprite と同じトリック)
- 描画は2段: (1) フレームに1回、N テクセルの `data` パスで位置/幅/色を
  書く(ADR-0014 のデータテクスチャ機構をそのまま流用)。(2) `triangle-strip`
  トポロジで **SEGMENTS=16 個の直線に分割した instanced ribbon** を直接
  ラスタライズし、最終画像(colorTex)に premultiplied-over で上描きする。
  march は一切発生しない
- フラグメントは辺のカバレッジ(`abs(cross) < 1` の範囲でのみ描く)で
  アンチエイリアスする。ブレンドは `srcFactor: one, dstFactor: one-minus-src-alpha`
  (標準の premultiplied over)

## 実装時に踏んだ罠(2つ)

1. **テクスチャキー解決の正規表現が `strip:` を通していなかった**
   (`/^((?:data|sprite):\d+):(\d+)$/` に `strip` を足し忘れ)。結果、頂点
   シェーダが常に 1×1 のダミーテクスチャ(ゼロ埋め)を読み、p0=p1=p2=(0,0)
   から接線がゼロベクトルになり `normalize` が NaN を返して**プリミティブが
   丸ごと画面外に消える**(完全な黒画面、エラーなし)という診断しづらい
   壊れ方をした。切り分け手順: (a) フラグメントを「常に白」に固定 →
   まだ真っ黒 → 頂点/ラスタライズ側の疑い、(b) 頂点位置を完全ハードコード
   → 表示された → パイプライン自体は正常、(c) 実際のワールド座標を色として
   出力するデバッグ varying を追加 → 依然ゼロ → **テクスチャ読み出しの手前**
   (パス経路・バインディング解決)を疑い、正規表現の不一致を発見。
   GPU デバッグは「入力が来ているか」を疑う前に「入力の**経路**が正しいか」を
   先に疑うべきという教訓
2. なし(1つのみ。上記が唯一かつ根本的な原因だった)

## Consequences

- ✅ 実測: vsync 解除の負荷試験で N=200〜20000 本の曲線において描画コストが
  ほぼ一定(~3800fps)。ADR-0014 のパーティクルと同じく、曲線数と描画コストが
  構造的に切り離された
- ✅ 幅が厳密(strip polygon なので march の踏み外し・段差アーティファクトがない)
- ✅ ADR-0015 の言語表面(`line a b`/`bezier a b c` に太さ引数を増やさず
  `outline` に委ねる)はそのまま維持される。ユーザーから見た構文・意味論は不変
- ⚠️ ADR-0002(Shape は常に SDF)からの局所的な逸脱: strip 化された line/bezier
  は距離場ではなくなる。`dist`/`grad` などの物理利用や `cut`/`inter` との
  組み合わせは対象外(元々 line/bezier の dist は非負なのでその用途は薄かった)
- ⚠️ **3D は未対応**(ユーザー判断でスコープ外。他の SDF 形状とのオクルージョンを
  正しく扱うには raymarch パスに深度出力を追加する必要があり、需要が見えたら
  再訪する。ADR-0013/0014 と同じ「拒否ではなく延期」の姿勢)
- ⚠️ ベジエは SEGMENTS=16 の直線近似(解析的な厳密形状ではなくなる)。曲率が
  非常に強い曲線や極端なズームでは近似誤差が見え得る(固定分割数なので
  適応的な分割は未実装)
