# ADR-0012: 入力は2つの正規形に固定し、拡張点を InputAdapter のみとする

- Status: accepted
- Date: 2026-07-04

## Context

音・マウス・MIDI に加え、TUIO(OSC/UDP のマルチタッチプロトコル)等の既存プロトコル
連携の要望が想定される。入力の種類ごとにコンパイラへ手を入れる設計は際限なく複雑化する。

## Decision

外部入力の正規形を2種類に固定する:

1. **スカラー uniform** — `mouse.x` `audio.lo` `midi.cc n`
2. **エンティティ表テクスチャ** — `fft`、TUIO カーソル、骨格、gamepad。
   「最大 N スロット+alive フラグ」の固定長 1D テクスチャ(静的要素数の言語制約と整合)

拡張点は `InputAdapter { name, schema, writeFrame }` の1インタフェースのみ。
コンパイラは schema しか見ず、新プロトコル対応でコンパイラ本体には一行も手が入らない。

- イベント(タッチ add/remove)は言語に持ち込まない。毎フレーム状態を標本化し、
  エッジ検出は時間族(`trigger`/`lag`/`age`)に任せる
- **汎用の JS ライブラリ FFI には踏み込まない**(スコープの線引き)
- OSC/UDP 系はブラウザが直接受けられないため UDP→WebSocket のローカル中継が必要。
  これはオプションのアダプタに閉じ込め、コア体験(URL を開けば動く)には影響させない

## Consequences

- ✅ TUIO は M5 の1アダプタにすぎず、設計は太らない
- ✅ 入力の種類がいくら増えてもコンパイラ・型システムは不変
- ⚠️ エンティティ数の上限 N はユーザーが意識する(静的要素数の帰結)
- ⚠️ ゼロインストールの例外が OSC 系にだけ生じる(ADR-0006 の但し書き)
