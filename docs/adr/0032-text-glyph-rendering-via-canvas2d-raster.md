# ADR-0032: `text` は Canvas2D でラスタライズした文字列を疑似SDFの Shape として扱う

- Status: accepted
- Date: 2026-07-07

## Context

創造的コーディング言語としての表現力を洗い出したところ、文字/タイポグラフィが
一切描けないことが分かった(フォント/グリフ関連のコードが皆無)。キネティック
タイポグラフィはライブビジュアルの定番表現の一つで、この欠落は実務上のギャップ。

対応にあたり2つの分岐があった:

1. **範囲**: 数字/ASCIIの単一グリフ(`glyph code`、既存の `grid`/`scatter`/`map`
   で複数文字を組む、言語仕様は不変)か、任意の文字列(日本語含む)をそのまま
   描けるようにするか。**後者を選んだ**。スコア表示のような動的な文字列を
   組み立てたい用途が多く、UIへ逃がす([[ADR-0031]] の HTTP URL と同じ手)は
   機能しないため
2. **文字列リテラル**: この言語には `"""..."""`(FFI用生文字列)しか文字列
   リテラルが無かった。1行文字列リテラル `"..."`(`\"`/`\\`/`\n` のエスケープ
   のみ)をレキサに追加し、既存の `str` AST/値表現(`VStr`)にそのまま乗せた
   (パーサ/型推論/staging は無変更で通った — 元々 FFI 用に `str` 型が
   汎用的に作られていたため)

## Decision

**`text h str`** を `circle r`/`box s` と同じ「サイズが最初の引数」の並びの
**Shape** として追加する(`fill`/`move`/`scale`/`glow`/`<+>` と自然に組み合わせる
ため。Field ではなく Shape にした)。

**ラスタライズはメインスレッドでのみ行う**: コンパイラは Worker で動く
(CLAUDE.md / worker.ts)ため DOM/Canvas2D に触れられない。`text` 呼び出しは
文字列内容のハッシュ(既存の `fnv1a`)をキーにして `ctx.textTextures` へ
`{key, text}` を登録するだけ(`blooms`/`stripBatches` と同じ「メタデータとして
コンパイル済みプログラムに持ち回る」パターン)。実際の
Canvas2D→`copyExternalImageToTexture` は `Renderer.evaluate()`(メインスレッド)
が行う。`fetch`/`getUserMedia` と違い `fillText` は同期APIなので、`await` 不要
で即座にテクスチャが用意できる(webcam/http のような非同期待ちが要らない)

**文字列ごとのテクスチャキャッシュ**(`TextTextureCache`、`PipelineCache` と
同じ LRU、上限64件)を新設。同じ文字列を毎フレーム焼き直さない

**実測アスペクト比の伝達**: 文字列の長さで幅が変わるので、コンパイラは
`${key}:aspect` という named input を `dist` 計算に埋め込むだけ(実測値は
知らない)。ランタイムは `px` と同じ「毎フレーム供給する入力」としてこれを
解決する(`Renderer.getInputFor` の default 分岐)。これにより **コンパイラは
DOM/Canvas2D に一切触れずに済む**

**疑似SDF**: ラスタライズ結果は真の距離場ではなくビットマップのアルファ値
なので、`dist = max(外接矩形のsdBox2, 0.5 - alpha)` という近似で組み立てる
(外接矩形の外側は sdBox2 が、内側は文字の塗り具合(alpha)が支配するよう
`max` で交差させる)。拡大するとぼやける(Lipschitz規律の対象外という割り切り、
[[ADR-0002]] 参照)

## Consequences

- ✅ `text 0.3 "SCORE 42" |> fill white |> glow 0.8 |> move [0, 0.3]` のように、
  既存の合成子へ何も特別扱いせず自然に接続できる(実機で確認済み: ASCII・
  日本語・fill/glow/move の組み合わせすべて正常動作)
- ✅ コンパイラ(Worker)は文字列とハッシュキーだけを知っていればよく、
  DOM/Canvas2D/GPUに一切触れない設計を維持できた
- ✅ 文字列リテラル追加は `str` という既存の値種別に相乗りしただけで、
  lexer 以外(parser/infer/stage)の変更はゼロだった
- ⚠️ ビットマップベースの疑似SDFなので、大きく拡大すると輪郭がぼやける
  (真のSDF/MSDFではない)
- ⚠️ フォントはブラウザの `sans-serif` 総称ファミリー固定(サイズ・ウェイト・
  フォント選択のカスタマイズは未対応、必要になれば `text` の引数を増やす形で
  拡張する)
- ⚠️ 複数行テキストは未対応(1行のみ)
