# ADR-0006: 実行環境はブラウザ+WebGPU、処理系は全部 TypeScript

- Status: accepted
- Date: 2026-07-04

## Context

Curv はデスクトップ OpenGL 3.3 依存が普及の足枷になった(ドライバ非互換、
Raspberry Pi 対応が WebGPU 移行待ち)。Punctual はブラウザ・ゼロインストールで
Estuary という共同ライブコーディング文化圏まで獲得した。

## Decision

- 実行環境は**ブラウザ+WebGPU**。作品のシェアが URL 一本になることを最優先
- 処理系(パーサ・型推論・Staging・WGSL 生成・ランタイム)は**全部 TypeScript** で
  ブラウザ内実行。コンパイラは毎評価走るが、ライブコーディングのプログラムは
  高々数百行なので JS で十分
- 将来ボトルネックが出たら Staging 以降だけ Rust/wasm 化する退路を残す

### 「実行速度で最適か」への回答(2026-07-04 追記)

実行速度の意味では最適ではないが、根拠のあるトレードオフである:

1. **ワークロードは高々数百行**。tsc の Go 移植や esbuild の速度差は数百万行規模で
   効く話で、この規模では 10 倍速くなっても体感差がない
2. **エンドツーエンドのボトルネックはドライバ内シェーダコンパイル**
   (`createShaderModule`)であり、実装言語を変えても縮まない。レイテンシ戦略の
   主戦場は「コンパイラを速くする」ではなく「再コンパイルを走らせない」
   (ADR-0008 の uniform 昇格、パス差分)で、そこはアーキテクチャで解決済み
3. 唯一の危険地帯は Staging の大規模ループ展開(`scatter 3000` で IR が百万ノード級)。
   これは言語切替より先に**「大きな N は展開せず WGSL ループとして出力する」**最適化で
   消すべき問題(シェーダ肥大対策としてもどのみち必要)

### 退路の具体化(同追記)

- **M0 からレイテンシ予算(implementation.md 7章)をベンチマークテストとして自動検証**する。
  推測ではなく計測で TS の限界を監視する
- Staging の入出力(型付き Core → Field IR)は純粋なデータ境界として保ち、
  **この段だけ後から Rust/wasm に差し替え可能**な状態を維持する。境界を跨ぐ
  API に JS 固有の構造(クロージャ等)を持ち込まない

## Consequences

- ✅ ゼロインストール。エディタ・コンパイラ・ランタイムが1ページに収まる
- ✅ 処理系自体の試行錯誤速度を優先できる
- ⚠️ WebGPU 非対応環境(古いブラウザ)を切り捨てる
- ⚠️ OSC/UDP 系入力(TUIO)だけはローカル中継が必要(ADR-0012 で例外として管理)
