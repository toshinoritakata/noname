# ADR-0028: ランタイムのホットスワップ経路を3点補強する(スクラブの流量制御・device.lost・パイプラインキャッシュの上限)

- Status: accepted
- Date: 2026-07-07

## Context

WebGPU/リアルタイムグラフィクスの観点でランタイム層(`src/runtime/`)を
読み直したところ、「編集し続けながら動かす」というこのツール特有の使い方
(ADR-0004 の2スロットクロスフェードが前提とする状況そのもの)でしか
顕在化しない問題が3つ見つかった。

1. **Alt+ドラッグのスクラブに流量制御がない**(`src/main.ts`)。
   `pointermove` は間引きなしで毎イベント `evaluate()` を呼んでいた。
   `evaluate()` はコンパイラ Worker への非同期往復を含み、Worker
   (`src/compiler/worker.ts`)はメッセージを到着順に同期処理する。
   結果はほぼ確実に「高速経路」(`programHash` 一致→ uniform 更新のみ、
   ADR-0008)に落ちるにも関わらず、そこに至るまで毎回 parse/infer/stage/
   WGSL生成のフルパイプラインを Worker で走らせていた。速いドラッグを
   すると Worker のメッセージキューに滞留が起き、画面の値がポインタに
   遅れて追従する — ADR-0008 が「<1フレーム」を謳うために作った当の
   デモ操作が、無throttleのせいで体感上壊れていた。
2. **`GPUDevice.lost` を一切観測していなかった**(`src/runtime/gpu.ts`)。
   ノートPCでの長時間ライブセット中にドライバリセットや電源管理でデバイスが
   失われると、`requestAnimationFrame` は回り続けるが実際には何も
   描画されなくなる。ADR-0010 の「エラー時は直前の正常プログラムが
   走り続ける」はシェーダコンパイル失敗のレベルでしか実装されておらず、
   デバイスレベルの障害は完全にカバー外だった。
3. **`PipelineCache` が無制限に肥大化する**(`src/runtime/program.ts`)。
   `ProgramSlot.destroy()` はテクスチャ/バッファは破棄するが、そのスロットが
   使っていたパイプラインをキャッシュから消す処理がなかった
   (キャッシュはビルド失敗時にしか縮まない)。ライブコーディングは
   「編集し続ける」のが前提の道具であり、構造的に異なる式を書くたびに
   `GPUShaderModule`/`GPURenderPipeline` がドライバ側に永続的に積み上がる。

## Decision

**1. スクラブの流量制御**: `main.ts` に「実行中の evaluate() があれば
次を1回だけ予約し、完了後にその時点の最新 `editor.value` で追いかける」
coalescing ラッパー `runEvaluate()` を追加し、scrub/デバウンス/Shift+Enter/
サンプル切替の全呼び出し箇所をこれ経由にした。Worker への postMessage 数
そのものは減らないが、キューに溜まる分をメインスレッド側で1件に合流させる。

**2. `device.lost` からの自動再初期化**: `Renderer` に `stop()`(内部 rAF
ループを止める)を追加。`main.ts` の GPU セットアップを `setupRenderer()`
として再利用可能な関数に切り出し、`device.lost`(`reason !== "destroyed"`)
で `stop()` →再初期化→直前のエディタ内容で再評価、を1回だけ自動で試みる。
GPU側の simulate/prev 状態は新デバイスに引き継げず失われるが
(ADR-0003 が元々「再初期化可能」を前提にした設計)、エディタのソースは
失われない。再初期化自体が失敗したら(その旨を)ステータス表示するだけに
留め、無限リトライはしない。

**3. パイプラインキャッシュの LRU 上限**: 素の `Map` だった
`PipelineCache` を、挿入順を保つ `Map` の性質を使った LRU 相当のクラスに
置き換え、上限(128エントリ)を超えたら最も参照されていないものから
捨てる。`ProgramSlot.build()` 側の `get`/`set`/`delete` 呼び出しは
インターフェースを変えていないため無変更。

## Consequences

- ✅ スクラブ中も Worker のキューが単調に伸び続けることはなくなる
  (常に「実行中1件+予約1件」が上限)
- ✅ デバイス消失時に画面が無言で固まる代わりに、状況がステータス表示され、
  自動的な復帰が試みられる
- ✅ 長時間の編集セッションでもパイプラインキャッシュがドライバ側の
  シェーダ資産を無制限に積み上げなくなる
- ✅ 実ブラウザ(Playwright、`channel: "chrome"` でシステムの Google Chrome を
  headless 起動。`navigator.gpu.requestAdapter()` が実アダプタを返すことを確認)
  で動作確認した:
  - スクラブの流量制御: Alt を押しながら60回の `pointermove` を約1秒で連射
    → 詰まりなく最終値に追従し、ドラッグ終了後は高速経路(`uniform 更新のみ
    0.4ms`)に収束
  - `device.lost` ハンドラ自体が例外を出さず配線されていること: 明示的な
    `device.destroy()`(`reason: "destroyed"`)を投げても設計通り no-op で
    既存の `Renderer` インスタンスを維持することを確認
  - `tsc --noEmit`/ビルド/既存 golden テスト(134件)も変更なしで通ることを確認した
  - 「このマシンには WebGPU デバイスがない」という当初の記述は誤りだった
    ([[ADR-0022]] も同様に修正済み)
- ⚠️ 未確認のまま残っているもの: (a) **本物の意図しない device.lost**
  (ドライバクラッシュ・電源管理等)からの再初期化——`device.destroy()` は
  常に `reason: "destroyed"` になり、この分岐(再初期化を試みる側)を
  ブラウザから安全に誘発する手段がなく、経路そのものはコードレビューでしか
  確認できていない。(b) LRU 上限(128エントリ)に実際に到達するほど
  長時間・多数の構造変更を伴うセッションでの検証
- ⚠️ device.lost からの再初期化は1回きりのベストエフォートで、連続して
  デバイスを失うような環境(仮想GPU、ドライバの深刻な不調等)では
  それでも復帰しない可能性がある。無限リトライや指数バックオフは
  今回は入れていない
- ⚠️ フェード中に再編集するとクロスフェードが完了前に打ち切られポップが
  起きる問題(2スロット設計の別の弱点)は今回のスコープに含めていない
  — 直すには ADR-0004 の2スロット制約自体を見直す必要があり、独立した
  設計判断として別途扱う
