# ADR-0004: ライブ遷移は `<>` クロスフェード、2スロット+BufferRegistry で実装する

- Status: accepted
- Date: 2026-07-04

## Context

ライブコーディングではコード再評価時の遷移品質が体験の核になる。Punctual は
`osc 440 >> audio <> 8` のように**遷移時間をコードで宣言する**構文を実証済み。
また、シェーダコンパイル中に映像が止まるとパフォーマンスが破綻する。

## Decision

- 言語構文: `out (...) <> dur` で再評価時のクロスフェード時間を宣言(省略時は即時切替)
- ランタイム: **ProgramSlot ×2**。新プログラムのコンパイル中も旧スロットが描画を続け、
  完了後に blend 係数を 0→1 に動かす。フレーム落ちなしが至上命題
- `simulate`/`prev` の実体テクスチャはスロットではなく **BufferRegistry** が所有する。
  「プログラムは使い捨て、状態は永続」という意味論をメモリ所有権にそのまま写す
- 時計は2本立て: `time`(リセットされない単調時計)と `etime`(評価時点起点、Punctual 継承)。
  フェード完了起点の `etime'` も併設して実験に委ねる

## Consequences

- ✅ 純粋関数(ADR-0001)だから新旧の同時評価+ブレンドが可能 — パラダイムの見せ場
- ✅ クロスフェード中の `prev` は blend 後の画像を書き戻すことで新旧どちらから見ても一貫する
- ⚠️ 一時的に新旧両方のパスが走るため、フェード中は GPU 負荷が最大2倍
- ⚠️ Registry の照合キー(束縛名+型)の設計が状態保持の成否を握る
