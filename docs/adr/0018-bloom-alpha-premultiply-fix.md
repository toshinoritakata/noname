# ADR-0018: bloom は glow のあるところでアルファも持ち上げる

- Status: accepted
- Date: 2026-07-06

## Context

`bloom`(画像合成・ポスト)は、ユーザーからのサンプルスクリプト依頼をきっかけに
実機で確認したところ、**シルエットの外側(黒背景)に一切にじまない**ことが
判明した。

診断の過程(実測、いずれも `range`/`sphere`/`render` を使った検証シーンで
ヘッドレス Chrome + ピクセル読み出しにより確認):
1. 4タップ・半径0.012(旧実装)では届く範囲が3px程度しかなく、シェーディング
   された3D図形のフチ(法線と光の角度が浅く暗い)からは、内側の明るい部分まで
   届かないと考え、3リング×8方向・半径0.03〜0.16まで拡張したが、それでも
   背景ピクセルの変化はゼロ(全250,634背景ピクセル中0件)のままだった
2. 半径を0.5(画面の約半分)まで極端に拡張しても、背景ピクセルは1件も変化
   しなかった。これはカーネル半径の問題ではなく構造的なバグだと判断
3. `image` パス自体の生成 WGSL の最終行を確認したところ、
   `return vec4f(c.rgb * c.a, c.a);` と**自分自身のアルファでRGBを事前乗算**
   していた(wgsl.ts)。一方 `brightPass` は
   `vec4f(max(c.rgb - 0.55, 0), 0.0)` と**アルファを常に0で返す**設計
   (元々は `<over>` 合成に影響を与えないための配慮)。この2つが組み合わさると、
   背景(アルファ0)にどれだけ明るい glow の RGB を加算しても、最終行の
   事前乗算で**必ずゼロに消える**——だから半径をいくら広げても効果ゼロだった

## Decision

`bloom` の出力アルファを、`base.a`(元画像の被覆率)と「このピクセルに乗って
いる glow の強さ」の **max** にする。glow の強さは加算する glow の RGB 最大
成分を 0..1 にクランプしたもの(グロー自体を可視化するのに十分な下限を
与えるだけで、被覆率としての意味は持たせない、簡易プロキシ)。

```
outRgb = base.rgb + q.rgb
glowAlpha = clamp(max(q.r, q.g, q.b), 0, 1)
outAlpha = max(base.a, glowAlpha)
```

これにより、`image` パスの `rgb * a` 事前乗算を通過しても glow の RGB が
消えずに残る。

同時に、カーネル自体も 4タップ・半径0.012 → 3リング(半径 0.03/0.08/0.16)
×8方向に拡張した(各リングは自分の方向数だけで平均し、リング間は加算する
実装 ─ 単純に全24タップを総重みで割ると遠いリングの寄与が近いリングの
タップ数に希釈されて弱くなりすぎるため)。

## Consequences

- ✅ 実測: 修正前は bloom on/off で背景ピクセルの変化が実質ゼロ(全画面
  250,634背景ピクセル中0〜数件、最大差0.0007程度)だったのが、修正後は
  全背景ピクセルに意味のある変化が現れ、実際の発光デモ(小さい明るい球)では
  各球の色に応じた光暈が黒背景へ視認できるレベルで滲むことを screenshot と
  ピクセル読み出しの両方で確認した
- ✅ `bloom` の言語表面(引数の型・意味)は無変更
- ⚠️ 大きい図形をシェーディングありで背景直前まで暗く落ちるように描画した
  場合(例: 半径0.6の球1個を画面いっぱいに)、フチから明るい中心部までの
  距離がカーネル半径(最大0.16ワールド単位)を超えることがあり、その場合は
  滲みが弱いままになる(実測: 30px reach が必要な場面でカーネルの実効到達が
  ぎりぎり足りず、変化量0.0001程度に留まった)。真のマルチパス(ダウンサンプル
  →ブラー→合成)ブルームではなく単一パス・固定半径の近似のため、これは
  今後カーネル半径や段数を増やせば緩和できるが、原理的な上限は残る
- ⚠️ `glowAlpha` のクランプ方法(rgb最大成分をそのままアルファ下限にする)は
  簡易的な近似。glow が非常に弱い(0.55をわずかに超える程度)場合、
  アルファも同程度弱くなるため、`<over>` で他の画像と重ねたときの見え方は
  厳密な物理ベースではない
