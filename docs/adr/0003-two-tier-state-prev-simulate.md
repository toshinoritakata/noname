# ADR-0003: 状態は prev / simulate の2層に限定する

- Status: accepted
- Date: 2026-07-04

## Context

純粋関数モデル(ADR-0001)ではトレイル・リアクション・ディフュージョン・相互作用する
パーティクルが書けない。しかし無制限のミュータブル状態を導入すると、ホットスワップの
安全性と GPU コンパイルの単純さが崩れる。

## Decision

状態の導入口を2つに限定する:

1. **`prev`** — 前フレームの最終出力画像を 2D 色場として参照する(Hydra の `src(o0)`、
   Lustre の `pre` と同型)。トレイル・無限ズーム用の最軽量の状態
2. **`simulate init f`** — 任意チャネルの場を「初期値+純粋な更新則」で進化させる。
   リアクション・ディフュージョンや力学系用の万能の状態

どちらも「ライブ書き換え時は**場の中身を保持して更新則だけ差し替える**」という
共通の意味論を持つ。

## Consequences

- ✅ GPU 実装がピンポンバッファに素直に対応する
- ✅ 閉形式で書ける表現(ステートレス・パーティクル、例10)と simulate の2層構造が
  言語の思想として明示される
- ✅ 更新則が純粋関数なので、状態を保ったままの規則差し替えが安全(ライブの見せ場)
- ⚠️ ランタイムに BufferRegistry(状態の所有者)が必要(ADR-0004)
- ⚠️ 状態の同一性は「束縛名+型シグネチャ」で決まる(名前を変えると新規扱い)と仕様に明記

## 物理シミュレーションの3段の切り分け(2026-07-04 追記)

本 ADR の2層が物理をどこまでカバーするかの整理:

- **第1段: 閉形式の運動(状態なし)** — 軌道が (シード, 経過時間) の閉形式で書ける
  運動(放物・軌道・ノイズ漂い)は純粋関数のまま数千個出る(例10)。「物理っぽい」
  表現の大半はここで足りる
- **第2段: gather 型の simulate — 設計済み** — 反応拡散・熱・波動(`laplacian`)、
  半ラグランジュ移流の流体、布・バネ格子、位置/速度テクスチャのパーティクル
  (オイラー/ベルレ積分)。`dt` 固定+サブステップで安定条件に手当。
  **SDF シーンがそのまま衝突ジオメトリになる**(`dist`/`grad` で貫通判定と反射法線、
  例12)のはこの設計の強いシナジー
- **第3段: scatter/ソート/アトミックが要る物理 — 意識的な範囲外** — 大規模 SPH
  (空間ハッシュ近傍探索)、拘束ソルバつき剛体。fragment ベース gather 専用の
  現 SimulatePass では書けない。拡張経路は SimulatePass の compute shader 化
  (implementation.md 4.2 の「将来 compute」)だが、純粋な更新則という言語表面に
  収まらないため、需要が実証されるまで延期(ADR-0013 と同じ「拒否ではなく延期」)
