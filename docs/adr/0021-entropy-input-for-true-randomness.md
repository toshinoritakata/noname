# ADR-0021: 真の乱数は `entropy` スカラー入力として注入する

- Status: accepted
- Date: 2026-07-07

## Context

`hash` は決定的な擬似乱数(入力の再ハッシュ)であり、作品関数の純粋性(ADR-0001)を
保つための設計だが、真にフレームごとに変化する乱数(`Math.random()` 相当)が欲しい
用途もある(パーティクルの初期散布、ノイズのシード替えなど)。作品関数の内部に
状態を持つ乱数源を持ち込むと、ホットスワップ・クロスフェード(ADR-0004)の前提が崩れる。

## Decision

真の乱数は既存の入力機構(ADR-0012: スカラー uniform)にそのまま乗せる。
`InputEngine.frame()` が毎フレーム `Math.random()` を `entropy` という名前のスカラー
uniform に書き込み(`src/runtime/inputs.ts`)、言語からは `entropy: Float` という
通常の入力として見える(`src/compiler/stdlib/inputs.ts`)。

- 作品関数のシグネチャは `(空間座標, 時間, 入力群) → 色` のままで、`entropy` はその
  一入力にすぎない。関数自体は純粋・決定的であり続ける(ADR-0001 は破らない)
- 乱数性の発生源はランタイム側(`InputEngine`)に閉じ込め、コンパイラ・型システムは
  `mouse.x` や `audio.lo` と同じ経路で扱う(ADR-0012 の正規形を追加種別で汚さない)
- 状態を伴う乱数(PRNG の内部カウンタを `simulate` 層に持たせる方式)は採らない。
  `prev`/`simulate` の2層(ADR-0003)は場の進化のためのものであり、
  ただの乱数のために状態管理の複雑さを持ち込む理由がない

## Consequences

- ✅ 実装コストが最小(値を1つ毎フレーム書くだけ。InputAdapter 拡張すら不要)
- ✅ 型システム・Staging・WGSL 生成は無変更(既存の `input` IR ノードがそのまま使える)
- ⚠️ クロスフェード中(ADR-0004)は新旧プログラムが同一フレームで異なる `entropy` 値を
  引く可能性がある。見た目のちらつきとして許容できるかは今後の実測で判断する
- ⚠️ `hash`(決定的)と `entropy`(非決定的)の使い分けはユーザーの理解に委ねる。
  リファレンスで明確に対比して説明する
