# ADR-0024: `bloom` の半径は独立引数ではなく `k` から適応的に決める

- Status: accepted
- Date: 2026-07-07

## Context

[[ADR-0023]] で `bloom k x` を `bloom k radius x` に変更し、ダウンサンプル段数
(滲みの実効半径)を独立引数として毎回指定させる案を実装した。しかし:

- 既存の `bloom k x` 呼び出し(例7・例15)がすべて破壊的変更の対象になる
- `radius` はコンパイル時定数(`grid` の次元と同様)である必要があり、
  「強さ `k` はオーディオ入力で毎フレーム変わるが、半径は固定」という
  非対称な制約をユーザーに説明・意識させる負担が増える
- そもそも `bloom` は元来 `Float -> Image -> Image` という他の画像ポスト
  (`fade`/`zoom`/`chromatic`/`grain`/`vignette`)と同じ形の1引数系だった

## Decision

`radius` の独立引数化を撤回し、`bloom k x` の2引数シグネチャに戻す。
その上で、ダウンサンプル段数を **`k` の静的な値から適応的に決める**
(`src/compiler/stdlib/postfx.ts`):

```
kStatic = k の静的値(sval)が分かればそれ、分からなければ 1 にフォールバック
levels  = clamp(round(3 + kStatic * 3), 2, 8)
```

- `k` が大きい(強く光らせたい)ほど段数を増やし、実効半径を広げる
- `k` が `0.3 + bass` のように動的な項を含み静的に決まらない場合
  (`ops.ts` の `foldConst` は両辺が静的でないと畳み込めない)は
  `kStatic = 1` とみなし、`levels = 6`(旧実装の固定値と同じ)にフォールバックする
- 段数はレンダーパス数を決めるコンパイル時の値である一方、`k` 自体は
  従来通りライブ更新可能な uniform のままで良い([[ADR-0008]])。両者を
  混同しないよう、段数の決定には `k` の**式評価結果ではなく静的値
  (`VNum.sval`)** だけを見る

## Consequences

- ✅ 既存コードとの互換性が高い(引数の数は変わらない。ただし段数の実際の値は
  `k` に依存して変わるため、生成される WGSL パス数はリテラルとしての `k` の
  値次第で変化する——同じ `bloom 1.2` でも本 ADR 前後で挙動は変わる)
- ✅ [[ADR-0023]] が課題としたモアレは、意図的に弱い(`k` が小さい)グロウを
  かけるケースで段数が自動的に減ることである程度緩和される
- ⚠️ `k` と半径を同じ数値に紐付けるのは見た目の直感(「強い光ほど滲みも広い」)
  に寄せた**ヒューリスティック**であり、物理的な必然性はない。今回の根本原因
  (グリッドの分割数とミップ解像度の一致)に対する厳密な修正ではなく、
  あくまで実用上の緩和策
- ⚠️ `k` が動的な式で静的に決まらない場合、ユーザーは段数を明示的に制御する
  手段を失う(常に既定値6にフォールバック)。将来的にこのケースを個別に
  指定したいという要望が出たら、[[ADR-0023]] のような独立引数を再検討する
