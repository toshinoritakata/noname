# ADR-0039: hash 系関数を bitcast+MurmurHash3(fmix32)ベースに置き換える

- Status: accepted
- Date: 2026-07-07

## Context

`hash`(WGSL側は `hash11`)を含む `hash11`/`hash12`/`hash13`/`hash21`/`hash22`
は、いわゆる「sin無しハッシュ」(Dave Hoskins系、`fract(n*0.1031+0.113)` を
非線形に混ぜる方式)で実装されていた。実測したところ、**入力が大きくなると
急激に衝突(同じ出力の重複)が増える欠陥**があった:

- `scatter N \i -> ... hash i ...` で N を増やすと、ユニークな出力数が
  頭打ちになる。N=1,000で98.2%、N=10,000で86.0%、N=50,000以降は
  **常に約12,025種類で完全に頭打ち**(N=100,000でもユニーク数は同じ12,025)
- `hash time` のように時間経過で使うパターンも、time が大きくなるほど
  劣化する。0.1刻み1000サンプルの窓で計測すると、time~0で98.4%
  ユニークだが、time~10,000(セッション開始から約2.7時間)で61.0%、
  time~100,000(約27時間)で19.5%まで低下する

原因は f32 の有効桁(仮数23bit、10進で約7桁)。`n * 0.1031` を計算する
最初の一手の時点で、n が大きいと積の下位の小数情報が失われ、異なる n が
同じ浮動小数点値に丸められてしまう。これは実装の細かいバグではなく、
「10進小数の乗算+fract」というアルゴリズムの構造そのものに起因する限界で、
定数をどういじっても本質的には解消しない。

## Decision

`hash11`/`hash12`/`hash13`/`hash21`/`hash22` の内部実装を、**入力の
IEEE754ビット列を直接 `bitcast<u32>` して整数ハッシュで混ぜる方式**に
置き換えた。整数ミキサーは MurmurHash3 の fmix32(既知の高品質アバランシュ
関数、shift-xor→乗算→shift-xor→乗算→shift-xor)をそのまま採用:

```wgsl
fn hashMix(seed: u32) -> u32 {
  var v = seed;
  v = v ^ (v >> 16u);
  v = v * 0x7feb352du;
  v = v ^ (v >> 15u);
  v = v * 0x846ca68bu;
  v = v ^ (v >> 16u);
  return v;
}
fn hash11(n: f32) -> f32 {
  return f32(hashMix(bitcast<u32>(n))) * (1.0 / 4294967296.0);
}
```

多次元版(`hash12`/`hash13`)は各成分を個別に `hashMix` してから
XOR で合成し、もう一段 `hashMix` する。複数出力版(`hash21`/`hash22`)は
一段目の結果に固定の salt(`0x68bc21eb` 等)を XOR してから再度
`hashMix` し、2つの出力を無相関にする。

**bitcast方式が効く理由**: 10進演算はどこかで「桁落ち」が起きうるが、
IEEE754のビット列は(NaN/±0 等の特殊値を除き)異なる浮動小数点値なら
必ず異なるビット列を持つ。ビット列をそのまま整数ハッシュに通すので、
入力の絶対値の大きさに関係なく、異なる入力は(アバランシュ特性により)
ほぼ独立に見える出力になる。

関数の名前・シグネチャ(引数の型・戻り値の型)は一切変更していないため、
`noise2d`/`noise3d`/`grainNoise`/`entropy` などの呼び出し側コードは
無変更で恩恵を受ける。CPU側インタプリタ(`interp.ts`、SDF性質テスト・
golden生成・エディタのインスペクタに使う)にも `Float32Array`/`Uint32Array`
経由の bitcast で同一アルゴリズムを実装し、GPU/CPU の結果を一致させた。

## Consequences

- ✅ 実測で確認: 整数0〜99,999の全域、および time 相当の大きな値
  (0〜1,000,000)で **衝突ゼロ(100%ユニーク)** になった(`test/hash.test.ts`
  で回帰テスト化)。旧実装で見えていた「大きいNのscatterや長時間セッションで
  パターンが頭打ちになる」問題が解消される
- ✅ 生成されるIRの呼び出し構造(`call hash11(...)`)は変わらないため、
  golden IRスナップショット(`test/golden/`)は無変更で通る。実際に変化する
  のは `hash11` 等の**WGSL関数の中身**だけ
- ✅ 実機(実ブラウザ+WebGPU)で確認済み: 既存サンプル(グリッドパターン・
  跳ねるパーティクル・TVノイズ・グリッチ)が引き続き正しく描画され、
  3000点規模の `scatter` + `hash` による散布が偏りなく分布することを確認
- ⚠️ 出力される具体的な数値列は旧実装と完全に変わる(同じ `hash i` でも
  違う値が返る)。ライブコーディング中に特定の `hash n` の見た目を
  「気に入っていた」場合、再コンパイル後に見た目が変わりうる
  (ただし本セッションの他のテストは exact な hash 値をアサートしていない
  ため、テスト面での破壊的変更はない)
- ⚠️ `grainNoise`(フィルムグレイン)のように `hash12` に渡す前の座標を
  自前で `time` 等でスケールしている箇所は、その**前処理の乗算自体**が
  依然として10進f32演算なので、そこだけ見れば長時間セッションでの精度
  劣化がゼロになったわけではない。ただし `hash` 系関数自体の頭打ちは
  解消されているため、影響は大幅に軽減される

## 関連

なし(hash系関数のアルゴリズム変更。呼び出し側のADRには影響しない)
