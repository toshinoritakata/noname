# ADR-0014: scatter の点状パーティクルはインスタンス化スプライトで描画する

- Status: accepted
- Date: 2026-07-05

## Context

`scatter N \i -> point r |> move pos |> glow ...` のような点状パーティクル(例10・12)は、
当初 `loopShape`(implementation.md 3.2-2)により CSG の min-union ループとして
レイマーチの `dist`/`colour` に組み込まれていた。これは ADR-0002(Shape は常に SDF)
に忠実だが、計算量が **O(ピクセル数 × march段数 × N)** になる。

ループ不変式の巻き上げ(座標に依存しない per-particle 計算をフレームに1回の
データテクスチャへ逃がす)と半解像度+段数削減を実装しても、「N 個の距離評価を
march 段ごとに行う」というループ構造自体は残り、N=300 で 13fps、N=4096 で 23fps
だった(実測、implementation.md 7章のレイテンシ予算とは別の実行時 FPS の問題)。

## Decision

`point |> move |> fill/glow` の連鎖(合成子が座標非依存の中心・半径・色だけを積む形)
を **静的に検出**し、該当する scatter は CSG ループを一切生成せず、代わりに:

1. 中心・半径・色をフレームに1回、N テクセルのデータテクスチャに書く(`data` パス、
   ループ不変式の巻き上げと同じ機構を流用)
2. **N インスタンスのビルボード描画**(`sprite` パス)でレイマーチの出力テクスチャに
   加算ブレンド・深度テストなしで重ね描きする

検出は `VShape.sprite` という軽量マーカーで行う: `point` が起点を置き、`move`
(3次元ベクトルのときのみ)/`fill`(場でない色のときのみ)/`glow` だけがこれを
引き継ぐ。`distort`/`warp`/`twist`/`repeat`/`scale` など座標を捻る合成子は
マーカーを引き継がない(= 安全に今まで通りの CSG ループへフォールバックする)。
`<+>` によるスプライトと通常 SDF の合成(例12: `world <+> scatter(...)`)は、
スプライト側の `dist` を定数 `+∞` にすり替えているため、既存の `min`/`select`
ロジックがそのまま正しく働く(実体のある側が必ず選ばれる)。

深度テストは行わない(加算ブレンドのみ)。カメラ背後の粒子はクリップ座標を
画角外に押し出して簡易カリングする。「壊れても絵になる」方向の割り切り
(ADR-0002 の Lipschitz 安全係数と同じ思想): 不透明な物体の背後にある粒子が
透けて見える場合があるが、視覚的な破綻は起きない。

**alpha は加算ではなく max でブレンドする**(2026-07-05 追記、実機バグ修正)。
rgb は加算(密集するほど明るく光る)で問題ないが、alpha を加算すると粒子が
密集した箇所で 1.0 を大きく超え、`<over>` 合成(`overBlend` の
`a = top.w + bot.w*(1-top.w)`)が `top.w` を被覆率(0..1)と前提しているため
式ごと破綻する。実機で確認した症状: colorTex の alpha が float16 の上限
65504 まで発散し、`(1-top.w)` が大きく負になって色が乱れる(ユーザー報告の
「レンダリングが不正」= 白い塊の中に斑点状のノイズ)。alpha は
`operation: "max"` にして 0..1 の被覆率のまま保つことで解決した。

## Consequences

- ✅ 実測 13fps→60fps(N=300)、23fps→60fps(N=4096)。vsync 解除の負荷試験では
  N=300〜20000 の範囲で描画コストがほぼ一定(~3000fps)——パーティクル数と
  描画コストが構造的に切り離された
- ✅ 検出は完全に既存のコンパイラパイプライン内で完結し、言語仕様・構文には
  一切変更がない(`scatter`/`point`/`move`/`fill`/`glow` の意味論は不変)
- ⚠️ ADR-0002(Shape は常に SDF)からの局所的な逸脱: 検出対象になった scatter は
  もはや「距離場」ではなく「描画のみのインスタンスリスト」になる。
  `dist world pos` のような物理利用(ADR-0003 の3段目、`grad`/衝突判定)は
  この経路を通らない(simulate 内の位置場は元々 state テクスチャの fetch で
  済んでおり、`loopShape` を経由しないため影響なし)
- ⚠️ 深度テストなし・加算ブレンドのため、不透明な物体の手前/奥の区別がない
  (常に「重ねて光る」)。物理的正確さより「速く・壊れても絵になる」を優先
- ⚠️ 検出対象は `point |> move(3D) |> fill/glow` の連鎖のみ。それ以外
  (`distort`・`warp`・`box`・場による着色など)は今まで通り CSG ループにフォール
  バックする(正しいが遅い)。需要が見えたら検出対象を広げる(拒否ではなく延期、
  ADR-0013 と同じ姿勢)
