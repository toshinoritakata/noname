# ADR-0019: bloom はダウンサンプル+ブラーの多パス連鎖にする

- Status: accepted
- Date: 2026-07-06

## Context

ADR-0018 で bloom のアルファ事前乗算バグを直した後も、ユーザーから
「グロウが花びら状(8角形のロブ)にムラになる」報告があった。原因は、
フル解像度で大半径(0.03〜0.16ワールド単位)・少方向(8〜16方向)の
離散タップでにじみを近似していたため、角度方向のエイリアシングが
目に見える形で出ていたこと。

ユーザーから「実際のレンダリング結果をダウンサンプルして合成するのが
普通では」という指摘があり、その通りだと判断した。実際のゲームエンジン
(Unreal/Unity 等)の bloom は、明るい部分を抽出→段階的にダウンサンプル→
各解像度で小さいカーネルのブラー→アップサンプルしながら加算、という
多解像度合成が標準。この方式は:
- 低解像度側の少ないタップ数が、フル解像度換算では広い実効ブラー半径に
  相当する(効率が良い)
- ダウンサンプル自体が平均化なので、角度方向のエイリアシングが出にくい
- 複数解像度の合成で自然な「明るい芯+柔らかい裾野」の光暈になる

## Decision

`bloom` を、単一パスの多方向タップ近似から、**5パスのダウンサンプル+
アップサンプル連鎖**に作り直す(`StagedProgram.blooms: BloomPassSpec[]`、
`ctx.blooms` として `render` 等と同様にパス列へ登録する形):

1. **extract**(半解像度): ユーザーの画像式を評価し `brightPass` を適用した
   結果を書き出す。ここだけがユーザーコードに依存する式(IR)
2. **down1**(1/4 解像度): extract を4タップのボックスフィルタで縮小
3. **down2**(1/8 解像度): down1 を同様に縮小
4. **up1**(1/4 解像度): down2 をバイリニアアップサンプル + down1(スキップ
   接続)を加算
5. **up0**(1/2 解像度、FINAL): up1 をアップサンプル + extract(スキップ
   接続)を加算。`bloom` の Field はこのテクスチャを1回サンプルして
   `base + k * サンプル` を返すだけ(以前と同じ「軽いテクスチャ読み出し」
   の形に戻る)

ダウンサンプル/アップサンプルの shader 自体はユーザーコードに依存しない
固定テンプレートなので、Codegen の IR emit を経由せず直接 WGSL 文字列を
組み立てる(`wgsl.ts` の `generateWGSL` 内、bloom セクション)。実行順は
`extract → down1 → down2 → up1 → up0`(depends on 順そのもの)で、
`image` パスより前に実行する(`program.ts`)。

**追記(2026-07-06)**: 「もっと広く発光させたい」との要望を受け、段数を
固定2段(1/4, 1/8)から `BLOOM_LEVELS = 4`(1/4, 1/8, 1/16, 1/32)に拡張。
ダウンサンプル/アップサンプルの生成をハードコードした個別呼び出しから
ループ(`downKey(i)`/`upKey(i)`)に一般化し、段数を変えるだけで実効ブラー
半径を調整できるようにした。実測: 段数を増やしてもレベルごとに解像度が
下がるため計算コストの増分は小さく、60fps・swap 8msを維持したまま光暈が
明確に広がることを確認した。IR レベルのシンボリック構造(`bloom:{id}:u0`
を1回サンプルするだけ)は変わらないため、golden snapshot にも影響しない。

**追記2(2026-07-06)**: 「もっともっと広げたい、HDR対応が必要では」という
追加要望・質問。確認したところ作業テクスチャは元々 `rgba16float`
(WORK_FORMAT、gpu.ts)で、1.0 を超える値もクランプされずに保持している
= 内部パイプラインは既に HDR。「広さ」の制約は HDR の有無ではなく
純粋にダウンサンプル段数だったため、`BLOOM_LEVELS` を 4→6(1/4〜1/128)に
拡張して対応。実測: 60fps・swap 26msを維持したまま光暈がさらに明確に
広がることを確認した。なお最終的な画面出力は `navigator.gpu.
getPreferredCanvasFormat()`(通常 `bgra8unorm`、標準SDR)にクランプされる
ため、真の HDR ディスプレイ出力(拡張色域キャンバス等)は別の、ブラウザ/OS/
モニタ依存の不確実性が大きいテーマとして未着手のまま残している。

**追記3(2026-07-06、バグ修正)**: 段数を増やした直後、ユーザーから
「Bloomが右に広がってしまう」と実機スクリーンショット付きで報告。各球の
光暈が丸ではなく右(または右下)に伸びた彗星状になっていた。原因は2つ:

1. **解像度の丸め不一致**: `program.ts`(CPU側)は各段のテクスチャを
   `Math.max(1, Math.floor(width / divisor))` で確保していたが、
   `wgsl.ts`(シェーダ側)は `U.header.xy / divisor` と**floor していな
   かった**。557 のような divisor(64, 128)で割り切れない解像度では、
   シェーダが想定する自分の解像度と実際にバインドされたテクスチャの
   ピクセル数がズレ、`uv = pos.xy / res` の変換が実寸より広く見積もられて
   微妙に歪む。段数が浅いうちは誤差が小さく目立たなかったが、
   `BLOOM_LEVELS` を6段に増やしたことで誤差が蓄積し、方向性を持った
   滲みとして可視化された
2. **ボックスフィルタのオフセット誤り**: ダウンサンプルの4タップオフセットを
   `0.5 / res`(dst 解像度基準)にしていたが、dst の1テクセルが src の
   2x2テクセルに対応する場合、src の隣接テクセル中心に届くには
   `0.25 / res` が正しい(`0.5/res` だと隣のブロックのテクセルへ飛んで
   しまう)。この誤りはオフセット自体は対称なので単独では方向性を持つ
   バイアスにはならないが、ぼやけ過ぎの原因にはなっていた

修正: シェーダ側の `res` 計算を `max(vec2f(1.0), floor(U.header.xy / divisor))`
にして CPU 側の確保ロジックと厳密に一致させ、ダウンサンプルのタップ
オフセットを `0.25 / res` に修正。実機で光暈が対称な円形に戻ったことを
screenshot で確認し、60fps・swap 11ms を維持したまま解決した。golden
snapshot への影響なし(symbolic IR は不変)。

**教訓3**: 「CPU側でテクスチャサイズを `floor` で決めているなら、GPU
シェーダ側で解像度を再計算する式にも必ず同じ `floor` を入れる」— 元の
解像度からの計算式が「数式としては同じ」でも、整数丸めのタイミングが
ズレるとサブピクセルの整合性が壊れる。多段パイプライン(ミップチェーン等)
ほど誤差が蓄積して可視化されやすいので、段数を増やす作業をしたときほど
この手の整合性を疑うべき。

**追記4(2026-07-06、切り分けと改善)**: さらにユーザーから「合成が不自然」
「bloomの形自体が歪んで見える」「addで合成されていないのでは」という報告。
実機で比較検証した結果、2つの独立した事象だと判明した:

1. **「二重丸/融合した泡状の形」は blendAll の仕様通りの挙動、bloomのバグ
   ではない**: 複数の球を近づけて配置したシーンで、`bloom` を外した版と
   付けた版を screenshot で直接比較したところ、**bloomなしでも全く同じ
   融合した泡状の形**が現れていた。これは `blendAll k`(smooth-min)が
   近接する球を実際に滑らかに繋げる、設計通りの幾何形状であり、bloom は
   その(すでに融合済みの)輪郭に沿ってグローを足しているだけだった
2. **アップサンプルのフィルタが単純すぎた点は実際に改善の余地があった**:
   ダウンサンプルは4タップの箱型フィルタだったのに対し、アップサンプルは
   1点だけの bilinear サンプルのみだった。非等方な箱型ダウンサンプルの
   影響がアップサンプル側で十分に均されず、光暈の輪郭がわずかに角ばって
   見える一因になっていた。Call of Duty の "next-gen post processing" 講演
   で示された標準的な3x3テントフィルタ(重み 1-2-1/2-4-2/1-2-1、9タップ)
   に置き換え、より等方的(丸い)なにじみになるよう修正した

実機で60fps・golden snapshot無変更を確認。切り分けの手順(bloomあり/なし
の同一シーンを screenshot で直接比較する)は、compositing の見た目の問題が
「合成そのもの」由来か「合成対象の形状」由来かを区別するのに有効だった。

## Consequences

- ✅ 実測: 花びら状のムラが解消され、滑らかな円形の光暈になった(実機
  スクリーンショットで確認)
- ✅ 実測: 大きい図形(半径0.6の球)でも全画面で最大輝度差5.92・背景の
  1839ピクセルが変化(旧実装は背景の変化が実質ゼロ〜数ピクセルだった)
- ✅ 60fps を維持(swap 10ms、実機計測)。ダウンサンプルするほど処理する
  ピクセル数が減るため、単一パス大半径タップ版より計算コストはむしろ低い
- ✅ `bloom` の言語表面(引数の型・意味)は無変更
- ⚠️ 固定2段階のダウンサンプル(半→1/4→1/8)なので、実効ブラー半径には
  上限がある。極端に広い光暈が欲しい場合は段数を増やす必要がある(今回は
  実測で十分な改善を確認できたため、まずはこの段数で様子を見る)
- ⚠️ 新しいパス種別(`bloom-extract`/`bloom-down`/`bloom-up`)と専用テクスチャ
  (`bloom:{id}:{e,d1,d2,u1,u0}`)を追加したため、`render`(`rm:N`)と同様に
  ProgramSlot が解像度変更時にこれらのテクスチャも再確保する
- ⚠️ `bloom` を複数回使うプログラムは、呼び出しごとに5パス×専用テクスチャが
  増える(現状 dream-code.md の使用例は1シーンにつき1箇所なので実害は薄い)
