# ADR-0029: OSC 入力を汎用ブリッジヘルパー(Rust)経由で追加する

- Status: accepted
- Date: 2026-07-07

## Context

ライブ演奏の実務観点でこのプロジェクトを見直したところ、TouchOSC/Lemur/
Resolume 等、VJ 現場の定番コントローラが話す**汎用 OSC**(TUIO のような
特定用途プロトコルではなく、任意のアドレスに任意の値を送る汎用メッセージ
形式)を受けられないことが分かった。ADR-0012 は TUIO のために
「OSC/UDP 系はブラウザが直接受けられないため UDP→WebSocket のローカル中継が
必要」と既に明記していたが、実際に中継サーバーを実装したことはなく
(`makeTuioAdapter` は WebSocket 接続先 URL を仮定するだけで、その先の
UDP→WebSocket 変換自体はこのリポジトリの外にある前提だった)、本 ADR が
このプロジェクト最初の中継ヘルパー実装になる。

**このタイミングで「いっそ全体を Rust でネイティブアプリに書き直すべきでは」
という提案も検討した。** 結論は不採用: 辛口レビューで挙がった欠落
(NDI/Syphon出力・汎用OSC・映像入力・マルチ出力・録画)は、どれも
「OS統合が必要な一部分だけを外部ヘルパーに切り出す」ことで解決でき、
ADR-0006 が明記する「ゼロインストール(URL を開けば動く)」という
このプロジェクトの核となる価値を壊す必要がない。フルネイティブ化は
言語処理系・ランタイム・エディタ UI を全部書き直すコストに見合わない
(ADR-0006/0013 が Curv の失敗から得た教訓を再確認しただけで終わった)。
**本 ADR はその代替として、「OS統合が要る入力/出力だけを、任意の言語で
書ける小さな常駐ブリッジヘルパーに切り出す」というパターンを、TUIO一件
限りの但し書きから、正式な拡張手段として一般化するもの**。

## Decision

`bridge/`(このリポジトリ内、Rust製)に OSC→WebSocket 中継ヘルパーを実装する:

- UDP でポート `9000`(OSC の一般的な既定ポート)を listen し、`rosc` crate で
  OSC メッセージをデコードする
- OSC アドレス文字列の**末尾の数字**をスロット番号とみなす(例:
  `/1/fader3` → スロット3、`/xy` のように数字がなければ無視)。
  最初の引数を f32 として読み、0〜31 の固定長配列(スロット)に書く
  (MIDI CC が 0〜127 の固定バンクであるのと同じ発想。任意アドレス文字列を
  そのまま言語に持ち込む案は、コンパイル時にソースを走査して
  `InputAdapter.schema` を動的に組み立てる必要があり、ADR-0012 の
  「schema はアダプタ登録時に一度だけ宣言する」という契約を拡張してしまう
  ため見送った)
- WebSocket でポート `3334` を listen し、スロット配列に変化があるたびに
  接続中の全クライアントへ現在値のスナップショット(32個の float の JSON
  配列)を送る。TUIO アダプタが「差分ではなく毎回スナップショットを送る」
  のと同じ設計に揃えた

ブラウザ側(`src/runtime/inputs.ts`)に `makeOscAdapter(url)` を
`makeTuioAdapter` と同じ形の `InputAdapter` として追加する。
`schema.scalars` に `osc.f0`〜`osc.f31` を宣言し、エンティティ表は使わない
(フェーダー/ノブの値だけなので、TUIO のような表構造は不要)。

言語側(`src/compiler/stdlib/inputs.ts`)に `osc.f n`(`midi.cc n` と同じ形)
を追加する。`n` は静的に決まる番号(0〜31)で、`osc.f${n}` という入力名に
解決する。

## Consequences

- ✅ TouchOSC/Lemur 等の汎用 OSC コントローラをフェーダーバンクとして
  使える(`osc.f 0` 〜 `osc.f 31`)
- ✅ ADR-0006 の「URL を開けば動く」コア体験は変わらない。ブリッジは
  OSC を使わない作品には一切不要な、完全にオプションの常駐プロセス
- ✅ 「OS統合が必要な部分だけをブリッジに切り出す」パターンが確立できたので、
  次に必要になった時(NDI 出力・Ableton Link 同期など)は同じ形(小さな
  常駐プロセス+WebSocket)で追加できる。今回は OSC だけに絞り、
  Ableton Link(C++ SDK ベースで Rust バインディングが未成熟)は
  別イテレーションに回した
- ⚠️ 任意の OSC アドレス文字列をそのまま言語に持ち込む柔軟性はない
  (番号付き固定バンクという割り切り)。`/1/fader3` のように末尾が数字の
  アドレスを送る前提のコントローラでないと使えない
- ⚠️ ADR-0012 が既に認めていた「ゼロインストールの例外は OSC 系にだけ生じる」
  を、実装を伴う形で確定させた(以前は明記だけで実体がなかった)
