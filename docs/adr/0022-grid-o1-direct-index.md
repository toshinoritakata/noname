# ADR-0022: `grid` の大 N はループではなく O(1) 直接索引にする

- Status: accepted
- Date: 2026-07-07

## Context

`grid`(`src/compiler/stdlib/iteration.ts`)は `UNROLL_LIMIT`(64)を超える N で、
`scatter`/`blendAll` と同じ「WGSL の for ループで全セルの距離を `min` 畳み込み」
方式(implementation.md 3.2-2、ADR-0014/0017 の踏襲)を使っていた。

しかし `grid` は `scatter` と違い、各セルが軸整列した固定サイズのタイルとして
規則的に並ぶ。クエリ点 `p` から「自分がどのセルに属するか」は `floor`/`clamp`
だけで一意に決まるため、本来は全セルを回して距離を比較する必要が一切ない。
実装を読み直して気づいた無駄な O(n) ループだった(ユーザーからの指摘がきっかけ)。

## Decision

`grid` の大 N 経路を、ループ+min畳み込みから直接索引に置き換える:

```
ix = clamp(floor((p.x + 1) / cellW), 0, nx - 1)
iy = clamp(floor((p.y + 1) / cellH), 0, ny - 1)
i  = iy * nx + ix
```

を計算し、そのセルの図形(`f i` の結果)だけをローカル座標で1回評価する。
WGSL の `loop`/`loopacc` ノードは一切生成しない。

前提: **各セルの図形は自セルの外にはみ出さない**(ADR-0012 と同様、
「タイル敷き詰め」という `grid` の用途を前提にした割り切り)。境界外の
クエリ点は最近傍セル(端の列/行)にクランプする。

## Consequences

- ✅ 大きなグリッドのフラグメントシェーダのコストが理論上 O(n) → O(1) になる
  (生成 WGSL からもループノードが消える)
- ✅ `scatter`/`blendAll` はハッシュ由来の不規則な配置(添字→座標が単射でない/
  逆算できない)なのでこの最適化は適用できず、引き続きループ化(ADR-0014/0017)
- ⚠️ セル境界を越えて隣接セルまで図形がはみ出す使い方(例: セル半径よりずっと
  大きい `circle` を並べる)をすると、全セル `min` 版とは結果が変わる
  (はみ出した部分がその隣接セルのクエリでは評価されなくなる)。
  そのような重なりが必要な場合は `scatter` を使うべき
- ⚠️ 実機 GPU でのベンチマークはまだ取っていない(このマシンには WebGPU
  デバイスがない)。IR/WGSL 出力レベルでループが消えたことと既存の golden
  テストでの構文検証は確認済み
