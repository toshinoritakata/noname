# ADR-0020: 最終合成に輝度ベースのReinhardトーンマッピングを挟む

- Status: accepted
- Date: 2026-07-06

## Context

作業テクスチャは `rgba16float`(HDR、ADR-0018/0019 参照)で、`bloom`/`glow`
などが 1.0 を超える値を作ることは意図された挙動。しかし最終的な画面出力は
`image` パスの `return vec4f(c.rgb * c.a, c.a);` で直接スワップチェーン
(標準 SDR、`bgra8unorm`)に書き込まれるため、1.0 を超えた値は**トーン
マッピングなしで単純にハードクリップ**されていた。ユーザーから「加色は
addで良いか、screenの方が良いか」という質問があり、検討の結果:
- ADD は物理的にも正しく HDR パイプラインとも整合する(ADR-0018/0019 で
  すでに議論済み)ので変更しない
- ただし ADD はクリップ前提の設計であり、`k` を大きくすると明るい部分が
  唐突に真っ白に飽和する。これを緩和するには screen ブレンドへの変更
  ではなく、**最終出力の直前にトーンマッピングを1段挟む**のが HDR
  パイプラインの設計として筋が良い、という結論になった

## Decision

`image` パス(最終2D合成)の最後、`c.rgb * c.a` で事前乗算する**直前**に、
輝度ベースの Reinhard トーンマッピングを挟む:

```wgsl
fn tonemapReinhard(c: vec3f) -> vec3f {
  let l = dot(c, vec3f(0.2126, 0.7152, 0.0722));
  return c / (1.0 + l);
}
```

チャネルごとではなく**輝度(luminance)でスケール**するため、明るい部分でも
色相・彩度が保たれる(チャネルごとの Reinhard だと非常に明るい色で色味が
薄まる/白っぽくなる)。`bloom` 固有の処理ではなく、`image` パス全体に
かけるユニバーサルな最終段(`glow` 等、bloom を使わない HDR 超過にも効く)。

## Consequences

- ✅ 実機で確認: 派手な色でも唐突な白飛びが緩和され、bloomの光暈も
  滑らかな階調になった
- ✅ 60fps 維持(全画面ピクセルあたり1回の dot積+割り算のみ、コストは
  無視できる)
- ✅ IR/staging には一切影響しないため、golden snapshot は無変更(この
  変換は WGSL コード生成層だけに存在する)
- ⚠️ `strip`(line/bezier の instanced 描画、ADR-0016)は `image` パスを
  経由せず最終画像に直接上描きするため、このトーンマッピングの対象外
  (通常は極端な HDR 値を持たないため実害は薄いが、線が非常に明るい色を
  持つ場合は将来的に検討の余地がある)
