# ADR-0023: `bloom` にコンパイル時定数の半径パラメータを追加する

- Status: superseded by [[ADR-0024]]
- Date: 2026-07-07

**この ADR の decision は撤回された**。半径を独立引数として毎回明示させる案は
API 破壊が大きく、ユーザーからのフィードバックで「既存の `k`(強さ)から適応的に
決めてみる」方針に差し替えた([[ADR-0024]] 参照)。Context(原因分析)の部分は
引き続き有効。

## Context

`bloom`(ADR-0019/0020)は、ダウンサンプル+アップサンプルの多解像度連鎖を
シーンの内容に関係なく常に**固定6段**で組んでいた。ユーザーが
`grid [5,5] \i -> box 0.1 |> fill (...)` のような「小さなハイライトが
規則的に並ぶ」シーンに `bloom` をかけたところ、本来は見えないはずの
グリッドの区切りが色の異なるベタ塗りブロックとして可視化される問題が発生した。

原因を CPU 側インタプリタ(`interp.ts`)で直接検証したところ、`bloom` なしの
画像は意図通り(セルごとに小さな点が1つだけ、他は完全に透明)であることを確認。
問題は `bloom` 側にあり、固定6段のダウンサンプル連鎖の最も粗いミップ(解像度
1/128)が、キャンバス解像度とグリッドの分割数(5×5)によってはミップの
1テクセルとグリッドの1セルがほぼ1対1に対応してしまい、そのテクセルの
(ハイライトの明るさで決まる)平均色がセルいっぱいに引き伸ばされて見える、
という「固定オンスクリーン解像度ベースの多重解像度ピラミッド」と
「シーン側の繰り返し構造」の間のモアレだった。

## Decision

`bloom k x`(`Float -> Image -> Image`)を `bloom k radius x`
(`Float -> Float -> Float(段数) -> Image -> Image` 相当。実際の型は
`Float -> Float -> Image -> Image`)に変更し、ダウンサンプル段数を
呼び出しごとにユーザーが指定できるようにする。

- `radius` は `grid` の `[nx, ny]` や `range n` の `n` と同様、
  **レンダーパス数を決めるコンパイル時定数**(`staticNum` で ASTリテラルの
  静的値を読む)。実行時に変化する値(`time` 依存など)を渡すとコンパイル
  エラーになる
- 内部の `BLOOM_LEVELS`(旧: 全 `bloom()` 呼び出し共通の固定値6)を、
  `bloom()` 呼び出しごとの `BloomPassSpec.levels` に変更(`src/compiler/wgsl.ts`)
- 段数を増やすほど実効ブラー半径が広がり、生成されるダウンサンプル/
  アップサンプルのレンダーパス数も比例して増える(段数 N → 2N+2 パス)
- 型推論(`infer.ts`)の `bloom` エントリも2引数版から3引数版に更新
- 既定値は用意していない(呼び出しごとに毎回指定が必要)。「シーンのスケールに
  対してどれくらいの半径が自然か」はシーンの内容次第で決め打ちできないため

## Consequences

- ✅ シーンのスケール(密なグリッドか、大きな単体のオブジェクトか)に応じて
  半径を調整でき、今回のようなモアレを avoid できる(小さい半径ほどミップの
  最粗レベルの解像度が高く保たれ、グリッドとの意図しない整合が起きにくい)
- ✅ 段数を減らせばレンダーパス数も減るため、副次的に軽量化にもなる
- ⚠️ 破壊的変更: 既存の `bloom k x` 呼び出し(例7・例15)は
  すべて `bloom k radius x` に書き換えが必要になった
- ⚠️ `radius` はコンパイル時定数なので、スクラブ(Alt+ドラッグ)や
  実行時アニメーションでは変えられない(形が変わる = 再コンパイル、
  ADR-0008 の高速経路の対象外)。動的に変えたい場合は再評価(Shift+Enter)
  相当のコストがかかる
