# ADR-0025: `bloom` の抽出はネイティブ解像度で評価し、そこから先はボックスフィルタのみで畳み込む

- Status: accepted
- Date: 2026-07-07

## Context

[[ADR-0023]]/[[ADR-0024]] は「密なグリッドに `bloom` をかけるとグリッドの
区切りがベタ塗りブロックとして可視化される」問題に対し、ダウンサンプル段数を
減らす(ADR-0023の手動指定 → ADR-0024の `k` からの適応)という緩和策を取った。
しかしこれは対症療法で、`k` を大きくして同じ密なグリッドに強い発光をかければ
同じ問題が再発する。

根本原因を捉え直すと、段数(mip階層の深さ)の話ではなく、**bloomの抽出
(extract)パスが最初から間違った解像度でサンプリングしている**ことだった。
`bloom` の実装は、レンダリング済みのテクスチャをダウンサンプルするのではなく、
シーンのSDF式を**もう一度、半解像度の規則的なサンプリング格子上で直接
評価し直して**いた(`img.fn(ctx, p2, span)`)。

これは「正則な格子でサンプリングする」ことと「正則な格子状の高周波コンテンツ
(グリッドの箱)を持つシーン」の組み合わせで、写真の網目やモアレと同種の
古典的エイリアシングを引き起こす。しかもシーン自身のアンチエイリアス
(`toImage` の `smoothstep(px, -px, d)`)は**ネイティブ解像度**を前提に
設計されているため、それを半解像度でサブサンプリングする時点で
サンプリング定理(Nyquist)に反しており、段数をいくら変えても
この最初の一歩の問題は直らない。

## Decision

bloomのダウンサンプル連鎖の先頭に、**ネイティブ解像度で1回だけユーザーの式を
評価する `native` パス**を追加する:

```
native(1/1, ユーザーの式を直接評価・brightPass 済み)
  → e(1/2, native からの2x2ボックスフィルタ)
  → d1(1/4) → d2(1/8) → … → dN(1/2^(N+1))   ※ 既存のボックスフィルタをそのまま流用
  → upN-1 … up0(既存のテントフィルタ+スキップ接続アップサンプル)
```

- `native` パスはシーンのアンチエイリアスが前提とする解像度(キャンバス
  ネイティブ)で評価するので、以前の「半解像度で直接評価」が持ち込んでいた
  エイリアシングの発生源そのものを取り除く
- `native → e` の変換は、既存の `downsample()`(2x2ボックスフィルタ、4タップ)
  をそのまま再利用するだけで実装できた(新しいフィルタ実装は不要)。以降の
  段(`e → d1 → … → dN`)も全部同じ実績のあるボックスフィルタ連鎖
- 3Dシーン(`render` 経由、`rm:0` テクスチャに一度レンダリング済み)の場合、
  `native` パスは実質「`rm:0` を評価してそのままコピーする」だけになり、
  追加コストはごく小さい。2Dシーンの場合は、以前は半解像度で行っていた
  シーン評価をネイティブ解像度で行うだけなので、計算量の増分は最大4倍
  (ただし1パスだけで、以降の段は全て解像度が下がっていくので全体への影響は
  小さい)
- `bloom k x` の言語シグネチャ・型・`k` から段数を適応的に決める仕組み
  ([[ADR-0024]])には変更なし。今回の修正は WGSL 生成(`src/compiler/wgsl.ts`)
  だけに閉じている

## Consequences

- ✅ 根本原因(半解像度での直接サブサンプリングによるNyquist違反)を解消。
  グリッドの分割数・キャンバス解像度・`k` の値の組み合わせに依存せず、
  同種のモアレが起きなくなる
- ✅ 既存の2x2ボックスフィルタ downsample 関数を1段追加で再利用しただけなので、
  実装・レビューの差分が小さい(新しいフィルタ設計・数式レビューが不要)
- ✅ [[ADR-0023]]/[[ADR-0024]] の「`k` から段数を適応させる」仕組みは
  引き続き有効(段数は依然としてブラー半径を制御する)。両者は独立した軸になった
  (半径=段数、アンチエイリアスの正しさ=このADR)
- ⚠️ レンダーパスが `bloom()` 呼び出しごとに1つ増える(native)。ネイティブ
  解像度のフラグメントシェーダなので、シーンの評価コストが軽くない場合
  (大きな `blendAll`/`grid` など)はその分の負荷が乗る
- ⚠️ 3Dシーンでも `rm:0` を経由せず直接 `img.fn` を評価する形は変えていない
  (`render` の結果はすでに `rm:0` サンプルに解決されるので、native パスは
  実質コピーになるだけで新規コストはほぼゼロ)
