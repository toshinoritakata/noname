# 先行研究メモ — Curv と Punctual

dream-code.md の設計(作品 = 時空間上の純粋関数、SDFシーン代数、2層の状態)に対する
2つの近縁言語の調査。何を採り、何を避けるかの判断材料。

---

## Curv — Doug Moen

数学でアートを作るための言語。主目的は 2D/3D の幾何モデリング(3Dプリント含む)。
純粋関数型・動的型付け。GitHub リポジトリは 2023-09 にアーカイブされ、
Codeberg (doug-moen/curv) に移行して継続中。Apache 2.0。

### Shape プロトコル(最重要の発見)

Curv の shape はレコード:

```
{ dist   : (x,y,z,t) -> 符号付き距離
, colour : (x,y,z,t) -> RGB
, bbox   : 軸平行バウンディングボックス
, is_2d, is_3d : bool
}
```

- **時間 t が dist と colour 両方の第4座標**。アニメーションが形状合成を複雑化させずに
  モジュラーに書ける。時間変換(加速・減速・ループ)も座標変換と同じ枠組み
- **dist と colour が1つのレコードに同居** — 我々の「図形と色の結婚問題」への回答。
  `fill` は colour フィールドを差し替える操作にすぎない、と整理できる
- SDF は近似でよいが Lipschitz 定数 1 以下という規律を課す(レイマーチの正しさの担保)

### シェイプ代数

プリミティブ群+ブール演算(union/intersection/difference)+滑らかブレンド+
剛体変換+非剛体変形(twist/bend)+反復(2D/3D タイリング)+ **2D↔3D 変換(押し出し・投影)**。
テクスチャは「強度場 (x,y,z,t)→0..1」×「カラーマップ 0..1→色」の合成として定義。

### GPU コンパイル戦略(SubCurv)

動的型付けの純粋関数型言語をこの手順でシェーダに落とす:

1. shape から dist / colour 関数を抽出
2. クロージャの捕捉変数をコンパイル時定数として**部分評価**
3. 関数呼び出しを**インライン展開**(再帰・多相を消す)
4. 静的型付けサブセット「GL (SubCurv)」に変換
5. OpenGL フラグメントシェーダを生成

→ 「言語全体は自由に、GPU に行く部分だけ静的なサブセットに絞る」という現実解。

### 弱点(我々にとっての差別化点)

- ライブコーディングは設計の中心ではない(書いて実行のサイクル)
- 音・外部入力なし
- 状態層(feedback / simulate)なし
- デスクトップ OpenGL 3.3 依存。WebGPU 移行は未完で、これが普及の足枷になった形跡
  (Raspberry Pi 対応が WebGPU 待ち、ドライバ非互換問題)

## Punctual — David Ogborn

ブラウザベース・インストール不要の**音と映像を1言語で**扱うライブコーディング言語。
Estuary(共同ライブコーディング環境)に組み込まれて使われる。
実装は PureScript + JavaScript、映像は WebGL、音は Web Audio。

### パラダイム

すべてが**シグナル(信号ネットワーク)**。`osc 440 >> audio` で音、
`[1,0,0] >> add` で映像レイヤに出力。出力先(add/blend/mul/rgb)が合成モードを兼ねる。

### 空間モデル

フラグメント座標 `fx` `fy` `fxy`(-1..1、中心原点)、極座標 `frt` `fr` `ft`。
変換は `zoom` `move` `spin` `tile` = 座標の再マップ(我々の warp と同型)。
低レベルには `setfx` `setfy` `setfxy` で任意の座標書き換え。

### 時間モデル(最重要の発見)

- テンポ中心: `cps`(cycles per second)、`time`(beat 0 からの秒)、`beat`
- **`etime` / `ebeat`: コード評価時点からの経過時間・拍** — 「評価した瞬間から始まる
  イントロアニメーション」がライブで書ける。グローバル時計と評価ローカル時計の2本立て
- `seq` : グラフの列を1サイクルに敷き詰める(Tidal 的パターンの入口)
- **`<>` : 再評価時のクロスフェード時間を宣言する演算子**。`osc 440 >> audio <> 8` は
  コード変更時に 8 秒かけて新旧をクロスフェード。
  → 我々の「ホットスワップ時クロスフェード」構想に言語構文レベルの先例がある

### 状態・入力

- `fb` : 前フレームの映像出力を参照(我々の `prev` と同じ、第1段階の状態のみ)
- 音解析: `lo` `mid` `hi` `fft`(出力の解析)、`ilo` `imid` `ihi` `ifft`(入力の解析)

### マルチチャネル意味論(採否要検討)

- デフォルトは**組合せ展開**: `[1,2] + [10,20]` → `[11,12,21,22]`(全組合せ)
- `+:` などコロン付きで**対応要素ごと**: `[1,2] +: [10,20]` → `[11,22]`
- 少ないコードで爆発的なバリエーションが出る反面、初見殺し。リスト展開は 64 要素上限

### 図形の扱い(我々との決定的な差)

`circle` `rect` `line` `chain` `mesh` 等はあるが、**返り値は 0/1 のマスク**(SDF ではない)。
`dist` `prox` で距離も取れるが、シェイプ代数(smooth union・モーフ・3D)は無い。
3D レイマーチングも無い。

### 摩擦点(ユーザー報告)

- 負数に括弧が要る(`(-1)*fx`)、空白区切り引数のパース曖昧さ、深いネストで括弧地獄
  — Haskell 風シンタックスをライブ用に流用したことの副作用
- 中間値の命名は `<<` 演算子(`l << [0, 0.1 .. 0.5]`)

## 採用・回避の判断

### Curv から採るもの
1. **Shape = {dist, colour} レコード + 時間第4座標** — 「図形と色の結婚問題」はこれで解決。
   `fill` は colour の差し替え、`morph` は dist と colour の同時補間として定義できる
2. **SubCurv 方式のコンパイル**(部分評価→インライン→静的サブセット→シェーダ)。
   ただし我々は最初から静的型+推論にして、このパイプラインを単純化する
3. Lipschitz 規律、2D↔3D 変換(押し出し・投影)、強度場×カラーマップの分解

### Punctual から採るもの
1. **`<>` クロスフェード宣言** — ホットスワップの遷移をユーザーがコードで指定する発想
2. **`etime`/`ebeat`(評価ローカル時計)** — グローバル `time` と併置する
3. tempo/cycle を言語組み込みにする姿勢、音解析入力の語彙(lo/mid/hi/fft)
4. ブラウザ・ゼロインストールという配布戦略(Curv の教訓の裏返し)

### 避けるもの
1. **組合せ展開のマルチチャネル意味論**(Punctual)— 面白いが初見殺し。
   明示的な `map`/`grid`/`scatter` で置き換える
2. **0/1 マスクの図形**(Punctual)— SDF を捨てるとモーフとブレンドが死ぬ
3. **デスクトップ GL 依存**(Curv)— 環境問題が普及を殺した実例として
4. Haskell 直輸入シンタックスのパース摩擦(負数・括弧)— ライブ用に構文を再設計する

### 確認できた我々のニッチ

「Pan の意味論 × SDF シーン代数 × テンポパターン × 2層状態 × ライブクロスフェード」を
全部持つ言語は存在しない。Curv は SDF 代数と時間第4座標を持つがライブと音がなく、
Punctual はライブと音を持つが SDF 代数と 3D と simulate がない。
両者の交差点+simulate 層が、この言語の固有領域。
