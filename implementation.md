# 処理系設計 — creative coding 言語(仮称)

dream-code.md の意味論と prior-art.md の教訓を実装に落とすための設計。
対象: 例1〜11 のすべてを最終的に支えられるアーキテクチャ。

---

## 0. 全体像

```
┌─ Editor (CodeMirror 6) ─────────────────────────────┐
│  Shift+Enter で評価 / 診断のインライン表示 / 数値スクラブ │
└──────────────┬──────────────────────────────────────┘
               ▼ ソーステキスト
┌─ Compiler(TypeScript、ブラウザ内で実行)────────────┐
│  1 Lexer / Parser        → AST(全ノードにソース位置)│
│  2 Desugar               → Core(パイプ・糖衣を展開) │
│  3 型推論                → 型付き Core               │
│  4 Staging(部分評価)   → Field IR(一階・純粋 DAG) │
│  5 パス分割              → RenderGraph(パスの DAG)  │
│  6 WGSL 生成             → パスごとのシェーダ+uniform表│
└──────────────┬──────────────────────────────────────┘
               ▼ RenderGraph + WGSL
┌─ Runtime(TypeScript + WebGPU)──────────────────────┐
│  ProgramSlot ×2(新旧プログラム、クロスフェード用)   │
│  BufferRegistry(simulate/prev の実体、スワップ跨ぎ保持)│
│  Clock(time / etime)・Inputs(audio / mouse / MIDI) │
│  フレームループ: uniform 更新 → パス実行 → blend → 画面│
└─────────────────────────────────────────────────────┘
```

**実装言語は全部 TypeScript**。理由: (a) ゼロインストール(ブラウザで完結)は Curv の
教訓(prior-art.md)から譲れない一線、(b) コンパイラは毎評価走るので JS で十分速い規模
(ライブコーディングのプログラムは高々数百行)、(c) 処理系自体の試行錯誤速度を優先。
将来ボトルネックが出たら Staging 以降だけ Rust/wasm 化する退路を残す。

## 1. フロントエンド

### 1.1 Lexer / Parser

- 手書きの再帰下降+演算子は Pratt パーサ(優先順位表で `|>` `<+>` `<over>` `<>` を管理)
- オフサイドルールは採らない。ブロックは式指向で `let ... in` と改行継続のみ
- **行頭 `|>` の省略**(打鍵コスト対策、'|>' 検討時の宿題): 前の行が式として未完で
  なく、次の行がインデントされていたら暗黙のパイプと解釈する。パーサレベルの糖衣
- 負数問題(Punctual の摩擦点)は単項マイナスを字句レベルで解決する:
  「`-` の直前がリテラル/識別子/閉じ括弧なら二項、それ以外は単項」
- **全 AST ノードがソース位置(span)を保持**。後段のどのエラーも必ず元コードの
  行列に写像する(シェーダコンパイルエラーまで含めて。→ 6.3)

### 1.2 エラー回復

パースエラーでも AST を返す(エラーノード挿入)。理由は2つ:
- 診断を複数出せる
- **評価は失敗しても映像は止めない**(TidalCycles 方式)。エラー時は直前の正常な
  プログラムが走り続け、エディタに診断だけ出る。これは言語仕様(dream-code.md)

## 2. 型システム

### 2.1 型の骨格

```
Scalar   ::= Float | Int | Bool
Vec n    ::= vec2 | vec3 | vec4
Point d  ::= vec2 (d=2) | vec3 (d=3)      -- d は次元変数
Field d a = Point d -> a                   -- 時間は暗黙の第4引数
Shape d   = { dist : Field d Float, colour : Field d Color }
Image     = Field 2 Color
Cam, Color(=vec4), Dur(時間リテラル用)
```

- **Hindley-Milner の制限版**: let 多相はあり、高階の型変数なし、再帰なし。
  推論は W アルゴリズムで十分
- **次元多相**: `move` や `<+>` は 2D/3D 両用。型変数 `d ∈ {2,3}` を持つ制限付き
  多相として推論し、**Staging で単相化**する(実行時に次元が混ざることはない)
- **スカラー昇格**: `Float` が `Field d Float` の要る場所に来たら定数場に自動リフト、
  `Float` と `Signal Float` の区別はユーザーに見せない(全部が暗黙にシグナル)。
  実装上は「係数の型に `time` 依存が含まれるか」を IR で追跡するだけ

### 2.2 エラーメッセージ

型注釈なしで書ける代償はエラーの質で払う(dream-code.md の宿題)。方針:
- 単一化失敗時は**期待側と実際側の「由来 span」を両方**示す
- ドメイン語彙で言い換える辞書を持つ(`Field 2 Color` → 「2Dの色場(Image)」)

## 3. Staging(部分評価)— この処理系の心臓部

Curv の SubCurv 方式(prior-art.md)を踏襲しつつ、最初から静的型なので単純化できる。

### 3.1 入出力

- 入力: 型付き Core(高階関数・let・レコードを含む)
- 出力: **Field IR** — 一階・純粋・静的単一代入の式 DAG。ノードは
  `p`(座標)/ `t`(時間)/ uniform 参照 / テクスチャサンプル / 算術・組み込み関数 /
  `select`(分岐は全部これ。GPU なので両側評価が基本)

### 3.2 変換規則

1. **インライン展開**: すべての関数適用を β 簡約。言語仕様で「クロージャは
   コンパイル時に展開可能」に制限してあるので必ず停止する
2. **ループ展開**: `grid [8,8] f` → f を 64 回展開して min/合成で畳む。
   `scatter n f` も同様。n は型検査後に定数畳み込みで確定していることを要求
   (できなければコンパイルエラー: 「要素数は静的に決まる必要があります」)。
   **大きな N(閾値目安 64 超)は展開せず WGSL の for ループとして出力する**:
   本体がインデックス以外に一様なら索引 i をループ変数に残せる。IR ノード爆発
   (JS 側の GC 圧)とシェーダ肥大の両方への対策で、言語を Rust 化するより先に打つ手
3. **単相化**: 次元変数 d を確定。2D/3D 両用の合成子はここで具体コードになる
4. **hash-consing**: IR ノードは構造的ハッシュで共有(共通部分式の除去を兼ねる)。
   `grid` 展開で生じる大量の同型部分式がここで潰れる

### 3.3 定数の uniform 化(ライブの隠し玉)

ソース上の**数値リテラルは定数畳み込みせず、原則 uniform に昇格**する。

- 効果: `circle 0.3` の `0.3` を `0.35` に書き換えたとき、IR の形が変わらないので
  **シェーダ再コンパイル不要、uniform 更新だけで即反映**(1フレーム以内)
- エディタ側で数値の上でドラッグ→リアルタイムに絵が変わる「スクラブ」が無料で手に入る
- 例外: ループ回数・グリッド数など構造に効く数は畳み込む(3.2-2)。
  「構造定数」か「値定数」かは IR 構築時に用途で自動判別する

### 3.4 パスの構造差分

IR は hash-consing 済みなので、**パス単位のハッシュ比較**で「前回とどこが変わったか」
が O(1) で分かる。再評価時:
- 形が同じパス → シェーダ再利用、uniform だけ更新(3.3 と合わせて最速経路)
- 形が変わったパス → そのパスだけ WGSL 再生成・再コンパイル

## 4. パス分割と RenderGraph

Field IR を GPU の実行単位に切る。切れ目は言語構造から機械的に決まる:

| 言語構造 | パス | 実体 |
|---|---|---|
| `simulate init f` | SimulatePass | ピンポンテクスチャ対 + 更新シェーダ(fragment で開始、将来 compute) |
| `prev` | 参照のみ | 最終出力の前フレームテクスチャを読む |
| `render cam shape3` | RaymarchPass | dist/colour を継ぎ足したレイマーチテンプレート |
| 2D の `out` 直下 | ImagePass | 全画面三角形 + 生成 fragment |
| `<over>` 連鎖・ポスト | ImagePass に融合 | 可能な限り1パスに融合(テクスチャ往復を減らす) |
| `<>` クロスフェード | BlendPass | 旧スロット出力と新スロット出力を mix |

- RenderGraph はパスの DAG。トポロジカル順に実行
- **パス融合**: `<over>` やポスト処理は中間テクスチャを作らず式として合成できる
  ケースが多い。「`prev`/`simulate`/`render` の境界だけが本当のパス境界」が原則

### 4.1 レイマーチャ(RaymarchPass の中身)

- スフィアトレーシングのテンプレート WGSL に、生成した `dist` / `colour` を継ぎ足す
- 法線は `dist` の有限差分(4サンプル)。`shade` 合成子はこの法線+ライト位置で
  ランバート+スペキュラを計算する式に展開される
- Lipschitz 違反対策(未解決課題の暫定解): 歩幅に安全係数 0.8 を掛けるオプションを
  デフォルト on。「壊れても絵になる」方向に倒し、規律は文書で促す

### 4.2 SimulatePass

- 状態は `rgba16float` あるいは `rgba32float` のテクスチャ対(ピンポン)
- 状態型はベクトルだけでなくレコードも許す(例12 の `{pos, vel}`)。コンパイラが
  チャネルにパック(4ch 超は複数テクスチャに分割)し、ユーザーには見せない
- `laplacian` 等の近傍演算は隣接テクセルのサンプルに展開。半ラグランジュ移流用に
  任意座標サンプル `sample s p` も提供(gather 型の範囲内)
- `dt` は uniform。固定タイムステップ(既定 1/60)+フレーム内サブステップ数の指定を許す

## 5. ランタイムとライブ機構

### 5.1 ProgramSlot ×2 とクロスフェード

```
slotA(現行)──┐
               ├─ BlendPass ── 画面
slotB(新)  ──┘
```

- 評価(Shift+Enter)でコンパイル開始。**完了するまで現行スロットが描画を続ける**
  (シェーダコンパイルは非同期 `createShaderModule` + `getCompilationInfo`)。
  フレーム落ちなしが至上命題
- 準備完了で `<> dur` に従い blend 係数を 0→1 に動かす。dur=0 なら即切替
- フェード完了後、旧スロットのリソースを解放(ただし BufferRegistry の中身は除く)

### 5.2 BufferRegistry — スワップを跨ぐ状態

`simulate` / `prev` の実体テクスチャはスロットではなく **Registry が所有**する。

- キーは「束縛名 + 型シグネチャ」(例: `rd : simulate vec2 @256x256`)
- 再評価時、新プログラムの simulate ノードを Registry に照合:
  - キー一致 → **テクスチャの中身を保持したまま更新則だけ差し替え**(言語の見せ場)
  - サイズ/型が変わった → 旧内容を新サイズにリサンプルして引き継ぐ(バイリニア)。
    引き継ぎ不能なら初期値から
  - 束縛名が変わった → 新規扱い(名前が状態の同一性、と仕様に明記)
- `prev` は Registry 内の「最終出力の前フレーム」1枚。クロスフェード中は
  blend 後の画像を書き戻す(新旧どちらのプログラムから見ても一貫した「前フレーム」になる)

### 5.3 Clock と Inputs

- `time`: パフォーマンス開始からの秒。**どんな操作でもリセットされない**単調時計
- `etime`: スロットごとに評価時刻を記録して `time - evalTime`。
  `<>` との相互作用(未解決課題)は「評価時点起点」を既定とし、
  `etime'`(フェード完了起点)を別名で提供して実験に委ねる
- audio: Web Audio `AnalyserNode`。`lo/mid/hi` は FFT ビンの帯域和を毎フレーム
  uniform に書く。`fft` はビン配列を 1D テクスチャで供給
- mouse / MIDI(Web MIDI): 同様に uniform 供給。すべて「ランタイムが毎フレーム
  書く uniform」に正規化されるので、コンパイラは入力の種類を区別しない

### 5.3.1 InputAdapter — 入力の拡張点(TUIO 等の既存プロトコル連携)

外部入力の正規形を **2種類に固定**する:

1. **スカラー uniform** — `mouse.x` `audio.lo` `midi.cc n`
2. **エンティティ表テクスチャ** — `fft`、TUIO カーソル、骨格トラッキング、gamepad。
   「最大 N スロット+alive フラグ」の固定長 1D テクスチャ。可変個数のエンティティを
   静的要素数の言語(3.2-2)に無理なく載せるための表現

拡張点はこのインタフェース1つだけ:

```typescript
interface InputAdapter {
  name: string                               // 言語から見える名前空間 (tuio.*)
  schema: ScalarSlot[] | EntityTable         // 型検査に使う宣言
  writeFrame(queue: GPUQueue): void          // 毎フレーム uniform/テクスチャに書く
}
```

- アダプタはランタイムに登録され、**コンパイラは schema しか見ない**。
  新プロトコル対応でパーサ・型推論・Staging・パス分割には一行も手が入らない
- 言語側の見え方(TUIO の例):

  ```
  dots = scatter 16 \i ->
           let c = tuio.cursor i        -- {pos, angle, vel, alive, age}
           in circle (0.08 * c.alive) |> move c.pos
  ```

  イベント(タッチの add/remove)は言語に持ち込まない。「毎フレーム状態を標本化」で
  一貫し、エッジ検出は時間族(`trigger`/`lag`/`age`)に任せる
- **ゼロインストールの例外を明示**: TUIO 等の OSC/UDP 系はブラウザが直接受けられず、
  UDP→WebSocket の小さなローカル中継が必要。これはオプションのアダプタに閉じ込め、
  コア体験(URL を開けば動く)には影響させない。Web MIDI / gamepad はブラウザ
  ネイティブなので中継不要
- **スコープの線引き**: 汎用の JS ライブラリ FFI には踏み込まない。入力は必ず
  InputAdapter(uniform/表への毎フレーム書き込み)に正規化できるものだけを受ける

### 5.4 tempo / cycle(例8系)

- `cps`(cycles per second)をランタイムが保持し、`cycle`/`every`/`1beat` は
  「`time * cps` を折り畳む式」に展開される(パターンはデータではなく式。
  Tidal のような無限パターン代数はスコープ外に置き、必要になったら再訪)

## 6. FFI(例11)

### 6.1 `wgsl (型) """..."""`

- 型注釈から関数シグネチャを機械生成(`Field 2 Float` → `fn(p: vec2f, t: f32) -> f32`)
- 中身はパースせず、名前だけリネームして(衝突回避)生成シェーダに継ぎ足す
- IR 上は「不透明ノード」。hash-consing はソース文字列のハッシュで参加

### 6.2 `glsl` / `shadertoy`

- naga(wasm ビルド)の GLSL フロントエンドで WGSL に変換してから 6.1 と同じ扱い
- `shadertoy` は `iTime/iResolution/iMouse/iChannel0` のシムを自動注入し、
  `mainImage` を `Image` 型の値にラップする
- naga-wasm はサイズが大きいので**遅延ロード**(初めて glsl ブロックに遭遇した時)

### 6.3 エラーの写像

生成 WGSL の各行に「由来 IR ノード → AST span」の対応表(ソースマップ)を持たせ、
`getCompilationInfo` の行番号をユーザーコードの位置に翻訳する。FFI ブロック内の
エラーはブロック内相対位置で示す。

## 7. レイテンシ予算(ライブの体感を数値で縛る)

| 経路 | 予算 | 手段 |
|---|---|---|
| 数値リテラルの変更 | < 16ms(1フレーム) | 3.3 uniform 化。再コンパイルなし |
| 構造の変更(1パス) | < 100ms | 3.4 差分再コンパイル+非同期コンパイル中も旧絵が出続ける |
| フル再コンパイル | < 300ms | それでも映像は途切れない(5.1) |
| パース+型推論+IR | < 20ms | プログラム規模が小さい前提。超えたら計測して手を打つ |

この予算は**M0 からベンチマークテストとして自動検証**する(CI で予算超過を fail に)。
TypeScript 実装の限界は推測ではなく計測で監視し、超えたら Staging 段のみ Rust/wasm 化
する(境界は「型付き Core → Field IR」の純粋データ。ADR-0006 の退路)。
なお実測上の支配項はドライバ内シェーダコンパイルであり、実装言語では縮まない——
主戦場は「再コンパイルを走らせない」(3.3 uniform 昇格、3.4 パス差分)側にある。

## 8. テスト戦略

- **IR スナップショット**: ソース → Field IR の文字列化を golden 比較(コンパイラの回帰)
- **ゴールデンイメージ**: 例1〜11 を headless WebGPU(deno/node + dawn か
  Chrome headless)で1フレーム描画し、画像差分
- **SDF 性質テスト**: プリミティブと合成子に対し「境界で ≈0」「Lipschitz ≤ 1+ε」を
  乱択サンプリングで検査(CPU 側に IR インタプリタを持つ — デバッグにも使える)
- **スワップテスト**: simulate の中身が更新則差し替え後も保持されることをバッファ読み戻しで確認

CPU 側 IR インタプリタは投資対効果が高い: 性質テスト・golden の期待値生成・
「この座標の値は?」というエディタ上のインスペクタ、の3役を兼ねる。

## 9. マイルストーン(垂直に切る)

| M | 内容 | 検証できる設計課題 |
|---|---|---|
| **M0** | 例1のみ: パーサ→型推論(単相)→WGSL→表示。Shift+Enter でホットスワップ+`<>` クロスフェード。定数 uniform 化 | ライブの体感、`out` 暗黙化の要否、レイテンシ予算 |
| M1 | 2D SDF 代数(プリミティブ・warp 族・`<+>`・morph)+ fill/`<over>` | Shape レコード設計、座標規約 |
| M2 | `grid`/`scatter` 展開+let 多相+次元多相 | Staging の停止性・展開コスト |
| M3 | 3D: `render`/カメラ/`shade`/レイマーチャ | パス分割、Lipschitz 安全係数 |
| M4 | `prev` + `simulate` + BufferRegistry | 状態保持スワップ(言語の見せ場) |
| M5 | audio/mouse/MIDI + tempo(`cycle`/`1beat`)+ InputAdapter 化(TUIO はここで1アダプタとして実装) | 入力の2正規形(スカラー/エンティティ表)の妥当性 |
| M6 | FFI(`wgsl` → `glsl`/`shadertoy`) | naga 遅延ロード、ソースマップ |

M0 が最重要。**M0 の時点でこの言語の存在理由(ライブの気持ちよさ)が検証できる**ように
切ってある。M0 で体感が出なければ設計に戻る。

## 10. 主要な設計判断の一覧(と根拠)

1. **全部 TypeScript・ブラウザ内コンパイル** — ゼロインストール(Curv の教訓)と
   試行錯誤速度。wasm 化は退路として保持
2. **定数リテラルの uniform 昇格** — 「数値をいじる」という最頻編集を再コンパイル
   ゼロにする。ライブ体感への投資対効果が最大の一手
3. **2スロット+Registry 分離** — 「プログラムは使い捨て、状態は永続」という
   意味論をそのままメモリ所有権に写した構造
4. **IR の hash-consing とパス差分** — CSE・差分コンパイル・テストの3つが
   1つの機構から出る
5. **CPU 側 IR インタプリタを持つ** — テスト・デバッグ・インスペクタの3役
6. **Tidal 級のパターン代数は初期スコープ外** — `cycle`/`every` は式展開で足りる。
   パターンをデータとして持つ設計は複雑度が跳ねるので、需要を見てから
7. **入力の正規形は「スカラー uniform+エンティティ表」の2種に固定、拡張点は
   InputAdapter のみ** — TUIO 等の既存プロトコルはアダプタ1個で済み、コンパイラは
